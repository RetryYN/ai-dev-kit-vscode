## Opus = オーケストレーター専任

| 委譲先 | 担当 |
|--------|------|
| Codex 5.3 | 設計（L2-L4固定）・コード実装（スコア8+）・コードレビュー |

Opus の仕事: 言語化、タスク分解、外注指示、出力レビュー・統合、エスカレーション判断。

### 禁止事項

以下を守ることで「委譲専念」を守る：

- ❌ **自分でコード実装・設計をしない** → Codex / Sonnet へ委譲（検証除く）
- ❌ **技術判断を独断しない** → TL（Codex 5.3）と壁打ちして決定
- ❌ **工程表を作ったのに毎回 PO 確認** → 例外条件（§工程表ベースの自律実行）に該当しない限り自律進行

## PM（Opus）から TL（Codex）への相談

PM（Opus）が技術的判断を TL（Codex 5.3）に相談する際のテンプレート。**PO（ユーザー）に技術的な質問を投げる前に、必ず TL に壁打ちする**。

### 相談テンプレート

```markdown
## TL相談: [判断内容の要約]

### 背景
[なぜこの判断が必要か / 現在の状況]

### 制約
[技術的制約 / ビジネス的制約 / リソース制約]

### 判断内容
[何を決める必要があるか]

### PM の仮説
[PM として考えている案とその理由 / 複数案がある場合は比較表]

### 質問
[TL に聞きたい具体的な質問]

### 受入条件
[どうなれば判断完了か / 何が得られればよいか]
```

### 相談すべきケース

| ケース | 例 | 相談先 |
|--------|----|----|
| 技術的判断 | 認証方式（OAuth/JWT）、DB選定、アーキテクチャ設計 | **TL（Codex 5.3）** |
| 実現可能性 | 「この工数で実装できるか」「技術的に可能か」 | **TL（Codex 5.3）** |
| 製品スコープ | 「この機能は必要か」「ユーザーは喜ぶか」 | **PO（ユーザー）** |
| ビジネス判断 | 価格設定、リリース時期、優先順位 | **PO（ユーザー）** |

### 禁止事項

- ❌ **技術的判断を PO に聞く**（「OAuth と JWT どちらにしますか？」など）
- ❌ **工程表作成後に毎回確認**（「タスク1完了しました。次に進めますか？」など）
- ❌ **TL に相談せず独断で技術判断**

### 工程表ベースの自律実行

工程表作成後は、以下の**例外条件に該当しない限り自律実行**する：

| 例外条件 | 対応 |
|---------|------|
| 技術的に実現不可能 | TL に相談 → PO にエスカレーション |
| 工数が見積もりの2倍超 | TL と再見積もり → PO に報告 |
| 要件の曖昧さ発見 | PO に確認（スコープ判断） |
| セキュリティ・個人情報関連 | PO に必ず確認 |

## プランモード TL 壁打ち

プランモード（EnterPlanMode）では、PO に質問・プラン提示する前に**技術的判断を TL に壁打ち**する。

### フロー

```
【Phase 1: 探索】→【Phase 2: 設計】→【Phase 3: TL壁打ち】→【Phase 4: PO確認】→【Phase 5: ExitPlanMode】
```

- **Phase 3（TL壁打ち）**: `codex exec` で TL に設計案をレビュー依頼
  - 技術的妥当性、代替案の有無、PO に聞くべき質問の精査
  - TL フィードバックを反映してからプランを確定する

### 対象/非対象

| 判断種別 | TL 壁打ち | PO 直行 |
|---------|----------|---------|
| 技術的判断（認証方式、DB選定、アーキテクチャ） | ✅ 必須 | - |
| 実現可能性（工数、技術制約） | ✅ 必須 | - |
| 製品スコープ（機能の要否、優先順位） | - | ✅ 直接 PO |
| ビジネス判断（価格、リリース時期） | - | ✅ 直接 PO |

## Agent 並列実行ルール

### 基本原則

- 並列は「相互依存がないタスク」に限定する（設計→実装 / 契約→実装 / 実装→検証 は順次）
- 「相互依存なし」の定義: 仕様/契約が確定済みで、同一ファイル/同一API/同一データセット（同一テーブル/同一外部データソース）を同時に変更しない
- 共有資源（同一ファイル/同一DBスキーマ/同一API/同一仕様）に触れる場合は順次
- 仕様・契約は「凍結」後に実装着手（凍結=TL合意済みのI/O・制約を ADR/設計書/README 等に明記）
- 依存が曖昧な場合は順次を優先し、必要なら先に"依存関係の確定"を単独で実行

### 並列実行の例

- **並列OK例**: 調査×2（独立した調査対象に限る） / テスト追加×2 / ドキュメント作成×実装レビュー / データ収集×要約 / 同一機能だが別モジュールへのテスト追加×2 / 独立したAPIエンドポイントの実装×2（契約確定済み）
- **順次必須例**: 同一ファイル編集 / 同一API契約の決定と実装 / スキーマ変更とクエリ実装 / 設計レビューと設計修正の往復

### 運用ルール

- 並列開始前に成果物仕様を統一（テンプレ/出力粒度/差分形式/レビュー観点）
- 並列時は成果物の形式/粒度を揃え、統合役（TL）が最終整合を取る
- 並列タスクで失敗/仕様不一致が出た場合は即停止→TLが原因切り分け→必要なら順次に切替
- リトライは「原因が特定できた場合のみ1回」、未解決なら順次化して再実行

### Task tool: resume の使い分け

- resume は「直前の結論や設計方針を引き継ぐ必要がある続き作業」に限定する
- 新規起動は「別テーマ／セカンドオピニオン／前回判断への疑義／コンテキスト肥大化」の場合
- 迷ったら新規、ただし修正対応や同一設計の継続検討は resume
- resume の過用はコンテキスト肥大化と矛盾蓄積の原因になるため注意

## Codex CLI

- 実装: `codex exec "プロンプト" -m gpt-5.3-codex`（5.2 の場合は `-m gpt-5.2-codex`）
- レビュー: `codex review --uncommitted`（常に 5.3）
- `--quiet` / `-q` は存在しない
- `--uncommitted` とプロンプト引数は併用不可
- **Codex 担当タスクを Sonnet で代替しない**。Task tool に Codex がなくても Codex CLI を使う

### レビュー手段の使い分け

| 成果物 | コマンド | 用途 |
|--------|---------|------|
| コード差分 | `codex review --uncommitted` | L4.5 実装ゲート(.2/.5)、L4.5→L4.7、L4.7→検証 |
| 設計書・仕様書 | `codex exec "レビュー: [対象]"` | L2→L2.5（設計レビュー）、L2.5→L3（API契約レビュー） |
| プラン・方針 | `codex exec "TL壁打ち: [内容]"` | プランモード Phase 3 |

## ADR（Architecture Decision Record）運用（ミニ）

- 対象: **後戻りコストが高い技術判断**（例: 認証方式、DB/ORM、外部API統合方針、データモデルの基幹設計、モノレポ/分割、権限制御）
- ルール: 重要判断は **TL（Codex 5.3）と協議**し、結論と理由を ADR に残す（"なぜそうしたか"が主目的）
- 保存先（プロジェクト優先）:
  - 第1候補: `./.claude/decisions/`（リポジトリ内）
  - 第2候補: `./docs/adr/`
- 最小フロー:
  1. PM が ADR をドラフト（Context / Decision / Consequences / Alternatives）
  2. TL がレビューして修正（必要なら agent-teams の ADR テンプレを使用）
  3. PO 判断が必要な影響（納期/コスト/UX/法務）がある場合のみ PO に確認
- テンプレ導線: `~/ai-dev-kit-vscode/skills/integration/agent-teams/references/team-presets.md` の「Preset C: ADR」テンプレを参照
