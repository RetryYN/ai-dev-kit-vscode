# 技術ストーリーテリングスキル（なぜ→何を→どうする→結果）

## 適用タイミング

このスキルは以下の場合に読み込む：
- README作成時
- ADR作成時
- 設計説明・提案書作成時
- 技術ブログ執筆時

---

## 1. 4部構成テンプレート

すべての技術文書を、次の順序で構成する。

```markdown
## なぜ（Why）
- 背景と課題
- 影響範囲（誰が困っているか）
- 変える必要性

## 何を（What）
- 解決方針の要約
- 対象範囲（in scope / out of scope）
- 成功条件（Doneの定義）

## どうする（How）
- 実現手段（設計・手順・構成）
- 代替案と不採用理由
- リスクと対策

## 結果（Outcome）
- 期待効果（定量/定性）
- 検証方法（指標・テスト）
- 次アクション
```

---

## 2. 各部の書き方ガイド（例文付き）

### なぜ（Why）

書き方の要点：
- 現状の不都合を具体化する
- 「誰に」「どの程度」影響するかを明示する
- 課題を1文で言い切る

例文：

```markdown
現状はデプロイごとに手動確認が必要で、平均30分の待ち時間が発生している。
このため、障害時の復旧開始が遅れ、運用チームの夜間負荷が高い。
```

### 何を（What）

書き方の要点：
- 解決策を抽象度高めに先出しする
- やらないことを明示して期待値をそろえる
- 受入基準を測定可能にする

例文：

```markdown
CIに自動スモークテストを追加し、デプロイ前の品質確認を自動化する。
今回は監視基盤の全面刷新は対象外とし、既存通知経路を利用する。
```

### どうする（How）

書き方の要点：
- 実装順序を番号で示す
- 代替案を1つ以上比較する
- 失敗時の切り戻しを定義する

例文：

```markdown
1. GitHub Actionsでstaging向けスモークテストを実行
2. 失敗時は本番デプロイを停止
3. ロールバック手順をRunbookに記載

代替案としてE2E全件実行を検討したが、実行時間が長くリリース待機が増えるため不採用とした。
```

### 結果（Outcome）

書き方の要点：
- 効果は指標とセットで記載する
- 検証タイミングを示す
- 次に何を判断するかまで書く

例文：

```markdown
デプロイ前確認の手作業を30分から5分へ短縮することを目標とする。
2週間運用し、失敗検知率とリリース遅延件数を比較して正式運用可否を判定する。
```

## 3. README向けストーリーテンプレート

```markdown
# プロジェクト名

## なぜ
- どの業務課題を解決するか
- 想定読者（利用者/運用者/開発者）

## 何を
- 提供機能の要約
- 非対応範囲

## どうする
- セットアップ手順
- 最小実行例
- トラブル時の確認手順

## 結果
- 導入後に得られる価値
- 成功判定指標（例: 作業時間、エラー率）
```

運用ルール：
- README冒頭の3段落で「なぜ」と「何を」を明示する
- コマンド列だけで終わらせず、最後に「結果」を記載する

## 4. ADR向けストーリーテンプレート

ADRの `Context → Decision → Consequences` を、ナラティブとして再構成する。

```markdown
# ADR-00X: タイトル

## なぜ（Context）
- 現状制約
- 解決したい課題
- 期限/運用条件

## 何を（Decision）
- 採用する意思決定
- 適用範囲
- 採用理由の要約

## どうする（Decision Detail）
- 実装方針
- 代替案比較（採用/不採用）
- 移行手順

## 結果（Consequences）
- 得られる利点
- 受け入れる欠点
- 監視指標と見直し条件
```

短文化例：

```markdown
Context: 既存APIはピーク時にタイムアウトが頻発する。
Decision: 読み取り系をキャッシュ層に分離する。
Consequences: 平均応答時間は改善するが、キャッシュ失効運用が追加で必要になる。
```

## 5. 設計提案書のストーリー構造

提案書は次の章構成を推奨する。

1. なぜ: 現状課題、機会、放置コスト
2. 何を: 提案概要、対象範囲、成功条件
3. どうする: アーキテクチャ、工程、体制、リスク対策
4. 結果: 期待効果、KPI、導入後の評価計画

最終ページに必ず記載：
- 意思決定してほしい項目
- 判断期限
- 判断に必要な追加情報

---

## 6. アンチパターン

| アンチパターン | 問題 | 修正方針 |
|---------------|------|---------|
| いきなりHowから始める | 背景不明で合意できない | 先にWhyを1段落で明示 |
| 結果が不明 | 成功失敗を判定できない | KPIと検証期間を書く |
| 課題が抽象的 | 優先度判断ができない | 影響対象と件数を記載 |
| 手段の羅列のみ | 何を解決するかが消える | Whatで目的を固定 |
| 代替案の不在 | 設計判断の妥当性を示せない | 最低1案比較を追加 |

---

## 7. 完了判定

- 4部構成（なぜ・何を・どうする・結果）が存在する
- 各セクションが前後で論理接続している
- 結果に検証可能な指標がある
- 文書末尾に次アクションが明示されている
