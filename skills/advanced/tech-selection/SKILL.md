# 技術選定スキル

## 適用タイミング

このスキルは以下の場合に読み込む：
- 新規プロジェクト技術選定時
- 技術スタック見直し時
- ライブラリ選定時

---

## 1. 技術選定の原則

### 基本方針

```
1. 要件から逆算する
   - 技術ありきで考えない
   - 解決したい問題は何か

2. 退屈な技術を選ぶ
   - 枯れた技術 > 最新技術
   - 実績 > 話題性

3. チームを考慮する
   - 学習コスト
   - 既存スキル
   - 採用市場

4. 長期視点で考える
   - 5年後も使えるか
   - メンテナンス性
   - コミュニティの活発さ
```

### 選定基準

| 観点 | 確認項目 |
|------|---------|
| 機能要件 | 必要な機能を満たせるか |
| 非機能要件 | 性能、セキュリティ、スケーラビリティ |
| 成熟度 | 安定性、バグの少なさ |
| コミュニティ | 活発さ、ドキュメント、サポート |
| 学習コスト | チームが習得できるか |
| 採用 | エンジニアを採用できるか |
| コスト | ライセンス、インフラ、運用 |
| 将来性 | 継続的に開発されるか |

## 2. 比較フレームワーク

### 評価マトリクス

```markdown
## 技術選定: データベース

| 観点 | 重み | PostgreSQL | MySQL | MongoDB |
|------|------|-----------|-------|---------|
| 機能要件 | 30% | 5 | 4 | 3 |
| 性能 | 20% | 4 | 5 | 4 |
| 運用性 | 20% | 4 | 5 | 3 |
| チームスキル | 15% | 5 | 4 | 2 |
| コスト | 15% | 5 | 5 | 4 |
| **合計** | 100% | **4.5** | **4.5** | **3.2** |

### 決定: PostgreSQL
理由:
- JSON対応が優れている
- チームに経験者が多い
- 拡張性が高い
```

### SWOT分析

```markdown
## Next.js SWOT分析

| Strengths（強み） | Weaknesses（弱み） |
|------------------|-------------------|
| SSR/SSG対応 | 学習曲線がある |

| 豊富なエコシステム | 設定が複雑になりがち |

| Opportunities（機会） | Threats（脅威） |
|---------------------|-----------------|
| App Routerの進化 | 競合フレームワーク |
| エッジ対応強化 | React自体の変化 |
```

## 4. 選定プロセス

### フロー

```
1. 要件定義
   └─ 機能要件・非機能要件の明確化

2. 候補リストアップ
   └─ 3-5個程度に絞る

3. 調査・評価
   └─ PoC実施、評価マトリクス作成

4. 比較・議論
   └─ チームで議論、トレードオフ確認

5. 決定・文書化
   └─ ADR作成、理由を記録

6. 検証
   └─ 小さく試す、問題があれば見直し
```

### PoC（Proof of Concept）

```markdown
## PoC計画書

### 目的
〇〇技術が要件を満たせるか検証する

### 検証項目
1. 基本機能の実装可否
2. パフォーマンス（目標: 〇〇ms以下）
3. 既存システムとの連携

### 期間
3日間

### 成果物
- サンプル実装
- 性能測定結果
- 評価レポート

### 判断基準
- 全検証項目をクリア → 採用
- 一部課題あり → 追加検証
- 重大な課題 → 不採用
```

## 5. ライブラリ選定基準

### npmパッケージ評価

```
✅ 選ぶべきサイン
- 週間DL 100万以上
- 直近3ヶ月以内に更新
- GitHubスター 5000以上
- TypeScript対応
- ドキュメント充実
- IssueのClose率が高い

⚠️ 注意サイン
- 最終更新が1年以上前
- IssueやPRが放置
- メンテナが1人だけ
- ドキュメントが不十分
- Breaking changeが多い
```

### 依存関係管理

```
1. 依存は最小限に
   - 本当に必要か検討
   - 自前実装のコストと比較

2. メジャーなものを選ぶ
   - ニッチなパッケージは避ける
   - フォークが必要にならないように

3. 定期的に更新
   - 月1回は依存関係を確認
   - セキュリティアップデートは即時

4. ロックファイルを管理
   - package-lock.json をコミット
   - 再現性を確保
```

## 7. 避けるべきアンチパターン

### 技術選定のNG

```
❌ 話題性だけで選ぶ
   「流行ってるから」は理由にならない

❌ 過剰なスペック
   将来必要「かもしれない」機能で選ばない

❌ 個人の好みで選ぶ
   チーム全体で使えるか

❌ ベンダーロックイン
   特定サービスに依存しすぎない

❌ 検証なしで採用
   必ずPoCで確認

❌ 記録を残さない
   後で「なぜこれを選んだ？」となる
```
