# AI Search スキル

## 使用タイミング（tavily との使い分け）

Perplexity Sonar 系は、検索結果の要約だけでなく仮説生成や論点整理が必要な場面で使う。

- G1R: 要件が曖昧で論点を仮説化したい
- R0: 大量ログ・仕様断片から仮説候補を整理したい
- R3: Intent Hypotheses のたたき台を素早く作りたい

### Tavily との基本方針

- `web-search (Tavily)`:
  - 一次情報の網羅収集を優先
  - URL 収集の精度と速度を重視
- `ai-search (Perplexity Sonar)`:
  - 推論付き整理、観点抽出、長文統合を重視
  - 仮説候補の生成と比較を重視

### 推奨シーケンス

1. Tavily で一次情報 URL を収集
2. Sonar Pro で論点統合と仮説化
3. 重要主張は一次情報で再検証

## 入出力契約

### 入力

- 調査テーマ: 何を明らかにしたいか
- 制約条件: 期間、地域、言語、対象業界
- 期待出力: 要約 / 仮説 / 比較表 / リスク一覧
- 重点: 日本語情報源優先の要否

### 出力

- 結論サマリ（3-7行）
- 推論 chain（根拠 → 解釈 → 仮説）
- 情報源リスト（最低3件、URL付き）
- 確度評価（高/中/低）
- 追加調査提案（不足根拠への次アクション）

### 推論 chain 形式

```markdown
推論 chain:
1. 観測事実: 公式発表で API 制限変更が告知
2. 解釈: 既存実装のスロットリング前提が破綻する可能性
3. 仮説: 短期的に 429 増加、再試行制御を先に改修すべき
4. 検証ポイント: 本番トラフィック分布と現行 retry 設定
```

## PERPLEXITY_API_KEY 設定

Perplexity MCP 利用には API キー設定が必要。

1. Perplexity 側で API キーを発行
2. 実行環境へ `PERPLEXITY_API_KEY` を設定
3. キーの平文保存とログ出力を禁止

```bash
export PERPLEXITY_API_KEY="<your_api_key>"
```

補足:
- CI 実行時は Secret 管理機構から注入する
- ローカルは `.env` またはシークレット管理ツールを使用する

## Sonar Pro と Sonar の違い

### Sonar Pro

- 深い推論・統合を優先
- 複数情報源の横断整理に強い
- 仮説生成、長尺調査、論点比較向け

### Sonar

- 低コスト・短時間応答を優先
- 軽量な事実確認や補助検索向け
- 即時性を優先する簡易調査向け

### 選択指針

- 仮説生成や設計判断材料が必要: Sonar Pro
- 単純な事実確認が中心: Sonar
- 迷う場合: まず Sonar Pro で骨子を作り、必要に応じて Sonar で追補

## 日本語精度メモ

日本語情報源を優先したい場合は、クエリ時点で明示する。

- クエリに `日本語`, `国内`, `JP` を含める
- 対象サイトや地域（例: `.go.jp`, `.or.jp`, 国内ベンダー公式）を指定
- 英語ソースのみで断定せず、日本語一次情報で補完する

### 実務上の注意

- 同じ論点でも日本語訳で意味がずれる可能性がある
- 法務・規制は必ず一次文書（原典）を確認する
- 日本語記事は公開日が古い場合があるため更新日を確認する

## 出力品質ルール

### 必須条件

- 単なる要約で終わらせず、推論 chain を明示する
- 情報源を 3件以上列挙する
- 断定箇所には根拠 URL を対応付ける
- 不確実な点は「仮説」として区別する

### 推奨条件

- 情報源を `公式/個人/二次` でラベル付けする
- 競合する情報がある場合は差分理由を記述する
- 時系列（いつ変わったか）を明示する

## 呼び出し例

### 例1: 要件曖昧時の仮説生成

- 入力: 「中小企業向け請求 SaaS の解約率を下げる一般的施策」
- 出力期待: 仮説 3案 + 根拠 + 検証指標

### 例2: 長尺調査の圧縮

- 入力: 複数レポートの URL 群 + 要約依頼
- 出力期待: 論点統合 + 相反点 + 意思決定候補

### 例3: 日本語情報優先

- 入力: 国内法規制に関する調査テーマ
- 出力期待: 日本語一次情報中心の根拠一覧 + 推論 chain
