# 階層間コントラクト定義

## 目的

HELIX フレームワークの各フェーズ間（L1→L2→L2.5→L3→L4→L4.5 実装→L4.7 Visual Refinement→検証→L5→受入）における以下を明確化：

- 入力/出力成果物
- PM（Opus）と TL（Codex 5.3 / Sonnet）の役割分担
- 検証内容・ゲート条件
- 失敗時の差し戻しフロー（逆流フィードバックループ）

## 用語の前提

SKILL_MAP.md のフェーズ名を採用：

- **フェーズ**: `L1` / `L2` / `L2.5` / `L3` / `L4` / `L4.5 実装` / `L4.7 Visual Refinement` / `検証` / `L5（デプロイ）` / `L6 受入`
- **レイヤー表記とフェーズ表記を混同しない**（本文書は階層間インターフェース定義）
- **検証フェーズは独立**（L2-L5 各層の検証を並列実施）

## PM/TL 役割分担

| 役割 | 担当者 | 責務 |
|------|--------|------|
| **PM（Project Manager）** | Opus | ゲート通過判定・スコープ管理・差し戻し判断・PO への報告・工程表作成（L4） |
| **TL（Tech Lead）** | Codex 5.3 | 設計・実装・技術検証・問題の特定・報告 |
| **TL（Test/Doc専任）** | Sonnet | テスト・ドキュメントのみ（実装禁止） |

### 役割詳細

| 役割 | PM（Opus） | TL（Codex 5.3） | TL（Sonnet） |
|------|-----------|----------------|-------------|
| **階層間遷移判断** | ✅ ゲート通過判定（3観点） | - | - |
| **スコープ管理** | ✅ 下方スコープ防止 | - | - |
| **技術的検証** | - | ✅ 各層の検証実行 | ✅ テスト・ドキュメント検証のみ |
| **成果物作成** | 工程表作成（L4） | 設計・実装 | テスト・ドキュメント |
| **問題発見時** | ✅ 差し戻し判断 | ✅ 問題の特定・報告 | ✅ テスト・ドキュメント問題の報告 |
| **PO への報告** | ✅ 受入判定・完了報告 | - | - |

## ゲート判定の3観点

PM が各階層完了時に判定する共通基準：

| 観点 | 判定内容 |
|------|---------|
| **① ユーザー要望充足** | PO の要求が全て成果物に含まれているか |
| **② 技術的課題クリア** | TL の技術検証が全て合格 |
| **③ 下方スコープなし** | 勝手に機能を削ったり簡略化していないか |

**3観点全て OK** → 次フェーズ進行
**1つでも NG** → 差し戻し（Issue 作成 → 修正 → 再検証）

## ミニレトロ（マイルストーン基準）

フェーズ間で学びを蓄積するスクラム的振り返り。固定レイヤーではなく**マイルストーン基準**で発火する（フェーズスキップに対応）。

### 発火タイミング

| マイルストーン | 発火タイミング | 目的 |
|--------------|--------------|------|
| **設計凍結後** | L2 完了後（API変更あり→L2.5後、外部依存あり→L3後） | 設計判断の妥当性振り返り |
| **実装凍結後** | L4.5 完了後（UI案件→L4.7後） | 実装の学び・工数乖離記録 |
| **終端フェーズ後** | L6 受入完了後（L6なし→検証完了後） | プロジェクト全体振り返り |

### フォーマット: ミニKPT

```yaml
mini_retro:
  phase: "設計凍結 | 実装凍結 | 終端"
  kpt:
    keep: "1行: 継続すべきプラクティス"
    problem: "1行: 問題・改善が必要な点"
    try:
      action: "1行: 次回試す改善策"
      owner: "担当モデル"       # 空文化防止: 必須
      due: "期限（次フェーズ開始まで等）"  # 空文化防止: 必須
  metrics:  # 実装凍結/終端のみ
    estimate_gap: "定性メモ: 見積もりと実績の乖離感（大幅超過/やや超過/適正/余裕あり）"
    rework_count: N  # 上位層差し戻し回数
  next_phase_input: "次フェーズへの申し送り事項"
```

### 運用ルール

- **実行者**: Sonnet（PM がレビュー）
- **ブロッカーにしない**: 原則 warning のみ。ただし以下の基準に該当する場合は blocking Issue 化:
  - 工数乖離が見積もりの2倍超（estimate_gap: 大幅超過）
  - rework_count が3回以上
  - verification §13 の即時エスカレーション条件と同一事象を検出
  - 判断主体: Sonnet が blocking 候補を提示 → PM（Opus）が blocking/warning を確定
- **記録先**: `docs/retro/YYYY-MM-DD-{milestone}-retro.md`
- **MEMORY.md 反映**: 終端レトロの `try` のうち cross-project 適用可能なものを追記
- **ゲート判定との関係**: 3観点合格後 → ④ミニレトロ記録 → 次フェーズ

## 事前調査ゲート（Web検索強制）

設計・実装前に先行事例を調査し、調査メモとして記録する。

### 強制条件

| 条件 | レベル |
|------|--------|
| 外部API連携（新規/変更） | **MUST** |
| 認証・認可ロジック | **MUST** |
| 新規ライブラリ/フレームワーク導入 | **MUST** |
| 技術選定（ADR対象） | **MUST** |
| メジャーアップグレード | **MUST** |
| 公開API/DB契約変更 | **MUST** |
| 決済・PII・法令影響 | **MUST** |
| 高負荷/並行性/移行（migration） | SHOULD |
| 純粋な内部リファクタリング/バグ修正 | 不要 |

### 挿入位置

```
L1完了 → ★事前調査ゲート → L2設計開始
L4完了 → ★実装.1の通過条件に事前調査を追加 → L4.5実装開始
```

### 実行フロー

```
PM: 強制条件に該当するか判定
├─ 該当 → Haiku 4.5 に調査タスク配送
│          └─ 調査レポート → 次フェーズの reference_docs に追加
├─ 非該当 → 「事前調査: スキップ（理由: ...）」を明示して進行
└─ 検索不可 → status: blocked（オフライン/API制限等）→ PM に報告
```

### 優先ソース（権威順）

| 優先度 | ソース | 信頼性 |
|--------|--------|--------|
| 1 | 公式ドキュメント / RFC（Context7 MCP 優先） | 最高 |
| 2 | GitHub Issues/Discussions（公式リポジトリ） | 高 |
| 3 | 技術ブログ（Qiita, Zenn, dev.to） | 中（要クロスチェック） |
| 4 | Stack Overflow | 中（回答の鮮度を確認） |

### 調査レポート出力

```yaml
research_report:
  topic: "調査対象"
  date: "YYYY-MM-DD"
  sources:
    - url: "URL"
      summary: "1行要約"
      relevance: high/medium/low
  conclusion: "設計/実装への影響"
  adoption: "採用/不採用の理由"
  risks: ["発見したリスク"]
```

**通過条件**: 「検索した」ではなく「調査メモが存在する」こと
**記録先**: `docs/research/YYYY-MM-DD-{topic}.md`

### 制約

- **機密情報の外部検索持ち出し禁止**: 検索クエリに社内固有名・API キー・顧客データを含めない
- **オフライン/検索不可時**: status: blocked として PM に報告（人間判断で代替手段を決定）

## 階層間コントラクト

### L1 → L2（要件 → 設計）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 要件リスト + 仮定一覧 | 設計書（画面・API・DB・セキュリティ） | 設計書作成 / 要件カバレッジ検証 | 要件を TL に伝達 / 成果物の完成度確認 / ゲート通過判定 + スコープ管理 | 全要件が設計書にトレース可能か | 3観点（①②③）適用 | L1 に差し戻し（要件の明確化）/ PM が PO に要件確認 |

### L2 → L2.5（設計 → API契約）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 設計書（API一覧・型定義・認証方式・エラーハンドリング） | OpenAPI 仕様 + TypeScript 型定義 | OpenAPI 仕様作成 / api-contract スキルで検証 / 不足情報・不整合を指摘 | 設計書を TL に伝達 / 成果物の完成度確認 / ゲート通過判定 + スコープ管理 / L2 修正指示（失敗時） | APIエンドポイント・型・認証方式の整合性 | 3観点（①②③）適用 + **`codex exec`（設計レビュー）合格** | L2 に Issue 返却 + 修正依頼 |

### L2.5 → L3（API契約 → 依存関係）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| OpenAPI 仕様 + 型定義 + DB設計 | 依存マップ + 実装順序 | 依存関係分析 / dependency-map スキルで検証 / API設計の問題点を指摘（失敗時） | 成果物を TL に伝達 / 実装順序の承認 / ゲート通過判定 + スコープ管理 / L2.5 修正指示（失敗時） | 循環依存の検出・実装可能性検証 | 3観点（①②③）適用 + **`codex exec`（API契約レビュー）合格** | L2.5 に差し戻し（API設計見直し） |

### L3 → L4（依存関係 → 工程表）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 依存マップ + 実装順序 | 工程表（タスクID・難易度・担当・スキル） | 技術的難易度の評価 / 技術的実現可能性検証 / 実装順序の問題点を指摘（失敗時） | 依存マップを受け取り / **工程表作成（PM の主要業務）** / 工程表の完全性確認 / ゲート通過判定 + スコープ管理 / L3 修正指示（失敗時） | 全タスクに割り当て完了・並列実行可能性 | 3観点（①②③）適用 | L3 に差し戻し（実装順序再検討） |

### L4 → L4.5 実装（工程表 → 実装）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 工程表 + 設計書 + API仕様 | コード + テスト + ドキュメント | 実装フェーズ（`implementation-gate.md` の実装.1〜.5 ゲート）/ codex review 実行 | 工程表に従って自律実行 / 進行管理 / 全タスク完了確認 | 全工程表タスクが完了しているか / codex review 合格 | 3観点（①②③）適用 | 該当タスク再実行（最大3回）→ L4 差し戻し（工程表見直し） |

### L4.5 実装 → L4.7 Visual Refinement（実装 → ビジュアル適用）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 実装コード + L2 ビジュアル方針 | デザイン適用済みコード | visual-design スキルに基づく配色・余白・タイポグラフィ・構図の適用 | ビジュアル方針の伝達 / 適用結果の確認 / 変更影響の分類（V0/V1/V2） | 4大原則・配色比率・視線誘導・連動設計の検証 | 3観点（①②③）適用 + **`codex review --uncommitted` 合格** | V0: L4.7内修正 / V1: L4.5差し戻し → L4.7再実行 / V2: L2差し戻し |

**変更影響の分類:**

| 分類 | 内容 | 対応 | 再実行上限 |
|------|------|------|-----------|
| **V0** 見た目のみ | 色・余白・タイポ・トークン変更 | L4.7 内で修正 → 検証へ | - |
| **V1** UI構造変更 | コンポーネント分割/レイアウト再編（API非変更） | L4.5 に差し戻し → 再実装 → L4.7 再実行 | 3回 |
| **V2** 契約影響 | API/DB/認可に波及 | L2（→L2.5→L3→L4→L4.5）→ L4.7 再走 | 初回: PM判断+PO報告必須 / 2回目: 人間エスカレ |

**スキップ条件:** 純バックエンド案件・UI変更なしの場合はスキップ可（SKILL_MAP.md フェーズスキップ決定木参照）

### L4.7 Visual Refinement → 検証（ビジュアル適用 → テスト検証）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| デザイン適用済みコード + テスト | 検証レポート（L2-L4.7 各層の検証結果） | 各層の技術的検証（quality-lv5 スキル使用）/ 問題点の特定 | 成果物を TL に伝達 / 検証レポート確認 / エスカレーション判断 | 設計整合性・API契約遵守・カバレッジ・品質・ビジュアル適合性 | 3観点（①②③）適用 + **`codex review --uncommitted` 合格** | レイヤー内ループ（最大5回）→ 上位層エスカレ |

### 検証 → L5（検証 → デプロイ）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 検証済みコード + 検証レポート | デプロイ結果 + パフォーマンステスト結果 | デプロイ実行 / セキュリティ・パフォーマンス検証 / 問題点の特定 | 成果物を TL に伝達 / デプロイ承認 / L5 修正指示（失敗時） | セキュリティスキャン・パフォーマンス基準 | 3観点（①②③）適用 | L5 に差し戻し（修正） |

### L5 → 受入（デプロイ → 受入）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 最終成果物 + 要件リスト（L1） | 受入判定レポート | 技術的な受入検証（verification §14 の観点で支援）/ verification スキルで検証 / 要件未達の特定 | 受入判定準備 / PO に報告 / PO に承認依頼 / エスカレーション | 要件リスト ↔ 最終成果物の突合 | 3観点（①②③）適用 + **PO 承認** | 該当レイヤーに差し戻し |

**最終承認**: **PO（人間）が最終承認**

## IIP（実装内割り込みプロトコル）

実装フェーズ（.2〜.5）で**前提崩壊や想定外の問題**を検出した場合の対応手順。通常の失敗（`failed`: 既知前提内のミス）とは区別する。

> 詳細（発動条件・前提/範囲見直し条件）は `implementation-gate.md` の IIP セクションを参照。
> 本節は**階層間インターフェースとしての分類・移送ルール**を定義する。

### 影響度分類と逆流ループとの関係

| 影響度 | 内容 | 対応 | 逆流ループ |
|--------|------|------|-----------|
| **P0**: ゲート内修正 | 同一ゲート内で解決可能 | 修正してゲート再判定 | 乗せない |
| **P1**: 実装内差戻し | 実装.1 の計画修正が必要 | 実装.1 に戻って計画修正→再進行 | 乗せない |
| **P2**: 上位層差戻し | 設計/契約の変更が必要 | 下記の逆流フィードバックループに移送 | **移送** |
| **P3**: 人間エスカレ | スコープ/前提の根本変更 | `verification §13` の人間エスカレ条件へ直行 | **直行** |

### `failed` vs `interrupted` の分離

| ステータス | 意味 | リトライカウント |
|-----------|------|----------------|
| `failed` | 既知前提内の実装ミス | **3回リトライ対象**（既存ルール） |
| `interrupted` | 前提崩壊/範囲再定義（IIP発動） | **リトライにカウントしない** |

### IIP の処理フロー

```
実装.2〜.5 で想定外の問題を検出
  ↓
TL: 影響度を分類（P0/P1/P2/P3）
  ↓
├─ P0 → 同一ゲート内で修正 → ゲート再判定
├─ P1 → status: interrupted → 実装.1 に差戻し（計画修正）→ 再進行
├─ P2 → status: interrupted → 逆流フィードバックループに移送
│        （L2/L2.5 に Issue 作成 → 修正 → 再検証 → 実装再開）
└─ P3 → status: interrupted → verification §13 人間エスカレへ直行
         （PM → PO に報告、人間判断を待つ）
```

**複数 IIP 同時発生時:**
- 最上位層への差し戻し（より上流の P2/P3）を**優先処理**
- 後発の IIP は status: `pending` として、先行 IIP 解決後に再分類
- 同一ゲートで P2 と P1 が同時発生 → P2 を先に処理（P1 は P2 の設計修正に吸収される可能性があるため）

## 逆流フィードバックループ

下位層で問題発見時の処理フロー：

| # | 責任者 | アクション | 成果物 |
|---|--------|---------|--------|
| 1 | TL | 問題検出（例: L2.5 で型定義に不整合） | 問題レポート |
| 2 | TL → PM | 報告「User 型の email フィールドが設計書にないが API仕様にある」 | - |
| 3 | PM | L2 に Issue 作成（TL に修正依頼） | Issue チケット |
| 4 | TL | L2 修正実施 | 修正済み設計書 |
| 5 | PM | L2 ゲート再検証（TL に依頼） | - |
| 6 | TL | L2 検証実行 | 検証レポート |
| 7 | PM | 合格判定 → "L2.5 再開" 指示 | - |
| 8 | TL | L2.5 作業再開（修正された設計書を使用） | OpenAPI 仕様 |
| 9 | PM | L2.5 ゲート検証（TL に依頼） | - |
| 10 | TL | L2.5 検証実行 | 検証レポート |
| 11 | PM | 合格判定 → "L3 開始" 指示 | - |

### P2（IIP 経由）の成果物管理

実装フェーズ（L4.5）から IIP P2 で逆流ループに移送された場合、実装済みコードの扱い:

1. **退避**: 実装成果物（.1〜現在のゲートまでの artifacts）を feature branch に退避（`git stash` または `git checkout -b wip/{task_id}`）
2. **設計修正**: 上位層（L2/L2.5）の修正を通常の逆流ループで完了
3. **再適用判定**: 設計修正完了後、PM が Codex 5.3 に diff レビューを依頼
   - 再適用可（設計変更と無矛盾）→ cherry-pick または merge
   - 再適用不可（設計変更と競合）→ 競合ファイルのみ再実装（実装.1 から再開）

### ループ不合格時のエスカレーション

| 条件 | PM の判断 | アクション |
|------|----------|---------|
| **5回ループ不合格** | 上位層に問題あり | L2 に差し戻し（設計見直し） |
| **技術的に実現不可能** | 根本的な問題 | L2 にエスカレ（アーキテクチャ変更） |
| **工数が見積もりの2倍超** | リソース問題 | PO に報告（スコープ削減 / リソース追加 / スケジュール延長） |
| **要件の曖昧さ発見** | スコープ問題 | PO に確認（L1 に戻る可能性） |
| **セキュリティ・個人情報** | 必須確認 | PO に必ず確認 |

## ゲート実装方法（実行例）

PM が TL に検証を依頼 + スコープ管理：

```bash
# 例: L2 完了時
PM: "L2 設計完了。L2.5 に進む前にゲート検証を実施"

# [PM → TL にゲート検証依頼]
PM: verification スキルを起動（L2 層検証）
TL: 技術的検証実行
  - 要件カバレッジ
  - API一覧の完全性
  - 認証方式の記載

# [TL から PM に検証結果]
TL: 検証結果を報告

# [PM がゲート判定: 3つの観点]
PM: ゲート判定
  ① ユーザー要望充足: PO の要求が全て設計書に含まれているか
  ② 技術的課題クリア: TL の検証が全て合格
  ③ 下方スコープなし: 勝手に機能を削ったり簡略化していないか

if ①②③ 全て OK; then
  PM: "L2.5 開始"
else
  PM: "L2 に差し戻し"
  PM → TL: Issue 作成
    「不足項目: 認証方式が設計書に記載されていない」
    「修正依頼: OAuth 2.0 / JWT のどちらを使うか設計書に明記してください」
  TL: 修正実施
  PM: ゲート再検証（合格まで）
fi
```

## ゲート判定チェックリスト（代表例）

### L2 設計 → L2.5 API契約

```markdown
## L2 → L2.5 ゲート判定

### ① ユーザー要望充足
- [ ] 全要件が設計書に記載されているか
- [ ] PO が求める機能が全て含まれているか
- [ ] 削除・簡略化した機能はないか

### ② 技術的課題クリア（TL 検証）
- [ ] API一覧が完全か（CRUD全て定義）
- [ ] 認証方式が記載されているか
- [ ] エラーハンドリングが設計されているか
- [ ] DB設計が完了しているか

### ③ 下方スコープなし
- [ ] 要件リスト（L1）と設計書（L2）が1:1対応しているか
- [ ] 勝手に「簡易版」にしていないか
- [ ] 「後で実装」で先送りした機能はないか

判定: [合格 / 要修正]
```

### L2.5 API契約 → L3 依存関係

```markdown
## L2.5 → L3 ゲート判定

### ① ユーザー要望充足
- [ ] 全APIが設計書のAPI一覧と一致しているか
- [ ] OpenAPI 仕様に記載漏れはないか

### ② 技術的課題クリア（TL 検証）
- [ ] 型定義が完全か（TypeScript 型 ↔ DB スキーマ整合）
- [ ] API ↔ 型 ↔ DB の3者が整合しているか
- [ ] 認証方式が実装可能か

### ③ 下方スコープなし
- [ ] 設計書（L2）のAPI一覧が全てOpenAPI化されているか
- [ ] 省略したエンドポイントはないか

判定: [合格 / 要修正]
```

## 例外パス（5回ループ不合格時）

### パターン1: レイヤー内で5回ループ不合格

```
L2.5 API契約フェーズ
  ↓
TL: OpenAPI 仕様生成 → PM: ゲート検証 → 不合格（1回目）
  ↓
TL: 修正 → PM: ゲート検証 → 不合格（2回目）
  ↓
（5回繰り返し）
  ↓
PM: エスカレーション判断
  ├─ 上位層に差し戻し（L2 に問題がある）
  │   ↓
  │   L2 修正 → L2.5 再開
  │
  └─ PO に報告（要件に問題がある）
      ↓
      PO: 要件明確化 → L1 から再開
```

### パターン2: 技術的に実現不可能

```
L3 依存関係フェーズ
  ↓
TL: 依存マップ生成
  ↓
TL → PM: 「循環依存が解消できない。設計の根本的な問題」
  ↓
PM: L2 に差し戻し（設計見直し）
  ↓
TL: L2 修正（アーキテクチャ変更）
  ↓
PM: L2 → L2.5 → L3 再実行
```

### パターン3: 人間エスカレーション

```
L4 工程表フェーズ
  ↓
PM: 工程表作成
  ↓
TL: 「この工数では実現不可能」
  ↓
PM: 工数見積もり再計算
  ↓
PM → PO: 「当初見積もりの2倍の工数が必要です」
  - 案A: スコープを削減（機能を減らす）
  - 案B: 追加リソース投入（外注・人員増）
  - 案C: スケジュール延長
  ↓
PO: 判断
  ↓
PM: 工程表再作成
```

## 逆流ループ終端ルール（無限ループ防止）

上記パターン1〜3のいずれにおいても、以下の終端ルールを適用する:

| 条件 | アクション |
|------|---------|
| 同一起点からの差し戻しが **2回目** | 1つ上位のフェーズへエスカレ（例: L2.5→L2 が2回目 → L1 へ） |
| **L2 まで到達**して再度5回不合格 | 人間エスカレ（PO へスコープ判断を委譲） |
| IIP P2 経由の逆流 | 差し戻し累積カウントに**含める**（interrupted はリトライカウント外だが、逆流回数は別管理） |
| L4/L4.5 間の差し戻しが **2回目** | L3 以上へエスカレ（工程表の前提を再検討） |

**差し戻し累積カウント**: PM がフェーズ間差し戻しの回数を追跡する。カウントはプロジェクト開始時にゼロ、上位フェーズへのエスカレーション成功時にリセットしない（累積管理）。
