# 階層間コントラクト定義

## 目的

HELIX フレームワークの各フェーズ間（L1→L2→L2.5→L3→L4→L4.5 実装→検証→L5→受入）における以下を明確化：

- 入力/出力成果物
- PM（Opus）と TL（Codex 5.3 / Sonnet）の役割分担
- 検証内容・ゲート条件
- 失敗時の差し戻しフロー（逆流フィードバックループ）

## 用語の前提

SKILL_MAP.md のフェーズ名を採用：

- **フェーズ**: `L1` / `L2` / `L2.5` / `L3` / `L4` / `L4.5 実装` / `検証` / `L5（デプロイ）` / `L6 受入`
- **レイヤー表記とフェーズ表記を混同しない**（本文書は階層間インターフェース定義）
- **検証フェーズは独立**（L2-L5 各層の検証を並列実施）

## PM/TL 役割分担

| 役割 | 担当者 | 責務 |
|------|--------|------|
| **PM（Project Manager）** | Opus | ゲート通過判定・スコープ管理・差し戻し判断・PO への報告・工程表作成（L4） |
| **TL（Tech Lead）** | Codex 5.3 | 設計・実装・技術検証・問題の特定・報告 |
| **TL（Test/Doc専任）** | Sonnet | テスト・ドキュメントのみ（実装禁止） |

### 役割詳細

| 役割 | PM（Opus） | TL（Codex 5.3） | TL（Sonnet） |
|------|-----------|----------------|-------------|
| **階層間遷移判断** | ✅ ゲート通過判定（3観点） | - | - |
| **スコープ管理** | ✅ 下方スコープ防止 | - | - |
| **技術的検証** | - | ✅ 各層の検証実行 | ✅ テスト・ドキュメント検証のみ |
| **成果物作成** | 工程表作成（L4） | 設計・実装 | テスト・ドキュメント |
| **問題発見時** | ✅ 差し戻し判断 | ✅ 問題の特定・報告 | ✅ テスト・ドキュメント問題の報告 |
| **PO への報告** | ✅ 受入判定・完了報告 | - | - |

## ゲート判定の3観点

PM が各階層完了時に判定する共通基準：

| 観点 | 判定内容 |
|------|---------|
| **① ユーザー要望充足** | PO の要求が全て成果物に含まれているか |
| **② 技術的課題クリア** | TL の技術検証が全て合格 |
| **③ 下方スコープなし** | 勝手に機能を削ったり簡略化していないか |

**3観点全て OK** → 次フェーズ進行
**1つでも NG** → 差し戻し（Issue 作成 → 修正 → 再検証）

## 階層間コントラクト

### L1 → L2（要件 → 設計）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 要件リスト + 仮定一覧 | 設計書（画面・API・DB・セキュリティ） | 設計書作成 / 要件カバレッジ検証 | 要件を TL に伝達 / 成果物の完成度確認 / ゲート通過判定 + スコープ管理 | 全要件が設計書にトレース可能か | 3観点（①②③）適用 | L1 に差し戻し（要件の明確化）/ PM が PO に要件確認 |

### L2 → L2.5（設計 → API契約）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 設計書（API一覧・型定義・認証方式・エラーハンドリング） | OpenAPI 仕様 + TypeScript 型定義 | OpenAPI 仕様作成 / api-contract スキルで検証 / 不足情報・不整合を指摘 | 設計書を TL に伝達 / 成果物の完成度確認 / ゲート通過判定 + スコープ管理 / L2 修正指示（失敗時） | APIエンドポイント・型・認証方式の整合性 | 3観点（①②③）適用 | L2 に Issue 返却 + 修正依頼 |

### L2.5 → L3（API契約 → 依存関係）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| OpenAPI 仕様 + 型定義 + DB設計 | 依存マップ + 実装順序 | 依存関係分析 / dependency-map スキルで検証 / API設計の問題点を指摘（失敗時） | 成果物を TL に伝達 / 実装順序の承認 / ゲート通過判定 + スコープ管理 / L2.5 修正指示（失敗時） | 循環依存の検出・実装可能性検証 | 3観点（①②③）適用 | L2.5 に差し戻し（API設計見直し） |

### L3 → L4（依存関係 → 工程表）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 依存マップ + 実装順序 | 工程表（タスクID・難易度・担当・スキル） | 技術的難易度の評価 / 技術的実現可能性検証 / 実装順序の問題点を指摘（失敗時） | 依存マップを受け取り / **工程表作成（PM の主要業務）** / 工程表の完全性確認 / ゲート通過判定 + スコープ管理 / L3 修正指示（失敗時） | 全タスクに割り当て完了・並列実行可能性 | 3観点（①②③）適用 | L3 に差し戻し（実装順序再検討） |

### L4 → L4.5 実装（工程表 → 実装）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 工程表 + 設計書 + API仕様 | コード + テスト + ドキュメント | 実装フェーズ（`implementation-gate.md` の実装.1〜.5 ゲート）/ codex review 実行 | 工程表に従って自律実行 / 進行管理 / 全タスク完了確認 | 全工程表タスクが完了しているか / codex review 合格 | 3観点（①②③）適用 | 該当タスク再実行（最大3回）→ L4 差し戻し（工程表見直し） |

### L4.5 実装 → 検証（実装 → テスト検証）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 実装コード + テスト | 検証レポート（L2-L4.5 各層の検証結果） | 各層の技術的検証（quality-lv5 スキル使用）/ 問題点の特定 | 成果物を TL に伝達 / 検証レポート確認 / エスカレーション判断 | 設計整合性・API契約遵守・カバレッジ・品質 | 3観点（①②③）適用 | レイヤー内ループ（最大5回）→ 上位層エスカレ |

### 検証 → L5（検証 → デプロイ）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 検証済みコード + 検証レポート | デプロイ結果 + パフォーマンステスト結果 | デプロイ実行 / セキュリティ・パフォーマンス検証 / 問題点の特定 | 成果物を TL に伝達 / デプロイ承認 / L5 修正指示（失敗時） | セキュリティスキャン・パフォーマンス基準 | 3観点（①②③）適用 | L5 に差し戻し（修正） |

### L5 → 受入（デプロイ → 受入）

| Input | Output | TL作業 | PM作業 | Verification | Gate | Failure |
|-------|--------|--------|--------|--------------|------|---------|
| 最終成果物 + 要件リスト（L1） | 受入判定レポート | 技術的な受入検証（verification §14 の観点で支援）/ verification スキルで検証 / 要件未達の特定 | 受入判定準備 / PO に報告 / PO に承認依頼 / エスカレーション | 要件リスト ↔ 最終成果物の突合 | 3観点（①②③）適用 + **PO 承認** | 該当レイヤーに差し戻し |

**最終承認**: **PO（人間）が最終承認**

## 逆流フィードバックループ

下位層で問題発見時の処理フロー：

| # | 責任者 | アクション | 成果物 |
|---|--------|---------|--------|
| 1 | TL | 問題検出（例: L2.5 で型定義に不整合） | 問題レポート |
| 2 | TL → PM | 報告「User 型の email フィールドが設計書にないが API仕様にある」 | - |
| 3 | PM | L2 に Issue 作成（TL に修正依頼） | Issue チケット |
| 4 | TL | L2 修正実施 | 修正済み設計書 |
| 5 | PM | L2 ゲート再検証（TL に依頼） | - |
| 6 | TL | L2 検証実行 | 検証レポート |
| 7 | PM | 合格判定 → "L2.5 再開" 指示 | - |
| 8 | TL | L2.5 作業再開（修正された設計書を使用） | OpenAPI 仕様 |
| 9 | PM | L2.5 ゲート検証（TL に依頼） | - |
| 10 | TL | L2.5 検証実行 | 検証レポート |
| 11 | PM | 合格判定 → "L3 開始" 指示 | - |

### ループ不合格時のエスカレーション

| 条件 | PM の判断 | アクション |
|------|----------|---------|
| **5回ループ不合格** | 上位層に問題あり | L2 に差し戻し（設計見直し） |
| **技術的に実現不可能** | 根本的な問題 | L2 にエスカレ（アーキテクチャ変更） |
| **工数が見積もりの2倍超** | リソース問題 | PO に報告（スコープ削減 / リソース追加 / スケジュール延長） |
| **要件の曖昧さ発見** | スコープ問題 | PO に確認（L1 に戻る可能性） |
| **セキュリティ・個人情報** | 必須確認 | PO に必ず確認 |

## ゲート実装方法（実行例）

PM が TL に検証を依頼 + スコープ管理：

```bash
# 例: L2 完了時
PM: "L2 設計完了。L2.5 に進む前にゲート検証を実施"

# [PM → TL にゲート検証依頼]
PM: verification スキルを起動（L2 層検証）
TL: 技術的検証実行
  - 要件カバレッジ
  - API一覧の完全性
  - 認証方式の記載

# [TL から PM に検証結果]
TL: 検証結果を報告

# [PM がゲート判定: 3つの観点]
PM: ゲート判定
  ① ユーザー要望充足: PO の要求が全て設計書に含まれているか
  ② 技術的課題クリア: TL の検証が全て合格
  ③ 下方スコープなし: 勝手に機能を削ったり簡略化していないか

if ①②③ 全て OK; then
  PM: "L2.5 開始"
else
  PM: "L2 に差し戻し"
  PM → TL: Issue 作成
    「不足項目: 認証方式が設計書に記載されていない」
    「修正依頼: OAuth 2.0 / JWT のどちらを使うか設計書に明記してください」
  TL: 修正実施
  PM: ゲート再検証（合格まで）
fi
```

## ゲート判定チェックリスト（代表例）

### L2 設計 → L2.5 API契約

```markdown
## L2 → L2.5 ゲート判定

### ① ユーザー要望充足
- [ ] 全要件が設計書に記載されているか
- [ ] PO が求める機能が全て含まれているか
- [ ] 削除・簡略化した機能はないか

### ② 技術的課題クリア（TL 検証）
- [ ] API一覧が完全か（CRUD全て定義）
- [ ] 認証方式が記載されているか
- [ ] エラーハンドリングが設計されているか
- [ ] DB設計が完了しているか

### ③ 下方スコープなし
- [ ] 要件リスト（L1）と設計書（L2）が1:1対応しているか
- [ ] 勝手に「簡易版」にしていないか
- [ ] 「後で実装」で先送りした機能はないか

判定: [合格 / 要修正]
```

### L2.5 API契約 → L3 依存関係

```markdown
## L2.5 → L3 ゲート判定

### ① ユーザー要望充足
- [ ] 全APIが設計書のAPI一覧と一致しているか
- [ ] OpenAPI 仕様に記載漏れはないか

### ② 技術的課題クリア（TL 検証）
- [ ] 型定義が完全か（TypeScript 型 ↔ DB スキーマ整合）
- [ ] API ↔ 型 ↔ DB の3者が整合しているか
- [ ] 認証方式が実装可能か

### ③ 下方スコープなし
- [ ] 設計書（L2）のAPI一覧が全てOpenAPI化されているか
- [ ] 省略したエンドポイントはないか

判定: [合格 / 要修正]
```

## ADR（意思決定ログ）の階層間統合

階層間の意思決定も ADR として記録：

```
docs/decisions/
  ├─ ADR-001-authentication-method.md（L2で決定）
  ├─ ADR-002-api-versioning.md（L2.5で決定）
  ├─ ADR-003-database-choice.md（L2で決定）
  └─ ADR-004-deployment-strategy.md（L5で決定）
```

### ADR と階層の対応

| 決定内容 | 決定する階層 | ADR例 |
|---------|------------|-------|
| 認証方式 | L2 | ADR-001-authentication-method.md |
| API バージョニング | L2.5 | ADR-002-api-versioning.md |
| DB選定 | L2 | ADR-003-database-choice.md |
| 依存関係解決 | L3 | ADR-004-dependency-resolution.md |
| デプロイ戦略 | L5 | ADR-005-deployment-strategy.md |

## 例外パス（5回ループ不合格時）

### パターン1: レイヤー内で5回ループ不合格

```
L2.5 API契約フェーズ
  ↓
TL: OpenAPI 仕様生成 → PM: ゲート検証 → 不合格（1回目）
  ↓
TL: 修正 → PM: ゲート検証 → 不合格（2回目）
  ↓
（5回繰り返し）
  ↓
PM: エスカレーション判断
  ├─ 上位層に差し戻し（L2 に問題がある）
  │   ↓
  │   L2 修正 → L2.5 再開
  │
  └─ PO に報告（要件に問題がある）
      ↓
      PO: 要件明確化 → L1 から再開
```

### パターン2: 技術的に実現不可能

```
L3 依存関係フェーズ
  ↓
TL: 依存マップ生成
  ↓
TL → PM: 「循環依存が解消できない。設計の根本的な問題」
  ↓
PM: L2 に差し戻し（設計見直し）
  ↓
TL: L2 修正（アーキテクチャ変更）
  ↓
PM: L2 → L2.5 → L3 再実行
```

### パターン3: 人間エスカレーション

```
L4 工程表フェーズ
  ↓
PM: 工程表作成
  ↓
TL: 「この工数では実現不可能」
  ↓
PM: 工数見積もり再計算
  ↓
PM → PO: 「当初見積もりの2倍の工数が必要です」
  - 案A: スコープを削減（機能を減らす）
  - 案B: 追加リソース投入（外注・人員増）
  - 案C: スケジュール延長
  ↓
PO: 判断
  ↓
PM: 工程表再作成
```
