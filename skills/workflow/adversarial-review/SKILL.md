# AI Adversarial Review スキル

## 適用タイミング

このスキルは以下の場合に読み込む：
- 設計判断で複数案がある時
- 本番環境への影響がある変更時
- 外部への説明責任がある決定時

---

## 1. 前提条件チェック

```
適用必須：
□ 本番環境への影響がある
□ ロールバックに1時間以上かかる
□ 外部（顧客・監査）への説明責任がある

スキップ可能：
□ 開発環境のみ＆5分で戻せる
□ 既存ルールの機械的適用
```

---

## 2. 三者分離アーキテクチャ

```
議長（Coordinator）
    │
    ├─→ Proposer A（独立インスタンス）
    │      └─ 独立コンテキスト
    │
    └─→ Proposer B（別モデル）
           └─ 独立コンテキスト
```

### 役割分離ルール

| 役割 | 責務 | 禁止事項 |
|------|------|----------|
| 議長 | 進行・裁定 | 自ら提案を出す |
| Proposer | 独立提案 | 他Proposerの出力を見る |

### 議長が提案者を兼ねる場合

1. 別インスタンスを起動（Task tool / codex exec）
2. 両提案者へのプロンプトは**同一文面**
3. 結果の要約は**引用ベース**（言い換え禁止）

---

## 3. コアサイクル

```
┌─────────────────────────────────────────┐
│  1. Propose（各AIが独立して案を作成）    │
│     ↓                                   │
│  2. Collide（差異・前提・評価軸を特定） │
│     ↓                                   │
│  3. Challenge（暫定結論への批判的検証） │
│     ↓                                   │
│  4. Decide（決定＋反対意見を記録）      │
└─────────────────────────────────────────┘
         ↑                    │
         └────────────────────┘
           批判で修正発生時（最大3回）
```

### 各ステップの出力

| ステップ | 出力 | 責任者 |
|----------|------|--------|
| Propose | 提案書（案＋根拠＋前提） | proposers |
| Collide | 差異表（同意点/相違点/理由） | coordinator |
| Challenge | 批判リスト（指摘＋代替案） | adversarial_reviewer |
| Decide | 決定書（結論＋反対意見＋根拠） | owner |

---

## 4. 独立性の確保

### 必須要件

- 異なるモデルプロバイダ（Anthropic/OpenAI）を使用
- 評価軸を2つ以上定義してから開始
- proposer同士は結論を共有しない
- 評価軸は議論開始前に固定・公開

### オプション強化

- 要約係を議長から分離
- 二重ブラインド化（モデル種別を伏せる）
- プロンプト凍結（途中変更不可）

### 利益相反の回避

- 自分が書いたコードのレビューは別AIが担当

---

## 5. 裁定基準（優先順位）

```
1. 倫理基準（法令・規約違反は絶対NG）
2. リスク最小化（被害規模を優先）
3. 可逆性（戻せる選択肢を優先）
4. ビジネス価値（同等なら価値が高い方）

競合時は上位基準が優先
```

---

## 6. 監査ログ

### 必須記録フィールド

```yaml
decision_record:
  decision_id: "ADR-YYYYMMDD-NNN"
  decision_summary: "何を決めたか"
  dissent_summary: "反対意見の要約"
  arbitration_basis: "裁定理由（どの基準を適用）"
  timestamp: "ISO 8601形式"
```

### 保管要件

- 保管場所: `.claude/memory/decisions/`
- 保管期間: 1年以上
- 形式: 第三者が追跡可能な構造化データ

### 追加ログ（推奨）

- プロンプト全文
- 各提案者の出力全文
- 要約・裁定の全文

---

## 7. 検証フィードバック

```
validation_loop:
  external: 実データ・専門家レビュー
  feedback: 検証結果は次回入力
  learning: 失敗パターン→anti_patternsに追加
```

---

## 8. 緊急時プロトコル

### 適用条件

- 障害対応など即時判断が必要
- 判断が24時間以内に可逆

### ガードレール

| 項目 | ルール |
|------|--------|
| 事後レビュー | 24時間以内に実施 |
| 月次監査 | 例外適用を集計 |
| エスカレーション | 月3回超→プロセス改善会議 |

---

## 9. チェックリスト

### 開始前

```
□ 適用条件を確認（必須/スキップ）
□ 評価軸を2つ以上定義
□ 提案者の独立性を確保
□ 議長と提案者を分離
```

### 進行中

```
□ 両提案者に同一プロンプトを送信
□ 結果は引用ベースで要約
□ 差異表を作成
```

### 決定時

```
□ 裁定基準の優先順位を適用
□ 反対意見を記録
□ 決定書を作成
□ .claude/memory/decisions/に保管
```

---

## 10. アンチパターン

| パターン | 問題 | 対策 |
|----------|------|------|
| 議長が提案者を兼ねる | 利益相反 | 別インスタンス起動 |
| 片方に有利なプロンプト | バイアス混入 | 同一文面を使用 |
| 評価軸の事後変更 | 恣意的裁定 | 事前固定・公開 |
| 反対意見の無視 | 盲点の温存 | 必ず記録・検討 |
| ログなし | 追跡不能 | 全文ログ保管 |
