# 見積もりスキル

## 適用タイミング

このスキルは以下の場合に読み込む：
- プロジェクト見積もり時
- タスク分解時
- スコープ調整時

---

## 1. 見積もりの原則

### 基本ルール

```
1. 見積もりは「予測」であり「約束」ではない
2. 不確実性が高いほどバッファを積む
3. 楽観値ではなく現実値で見積もる
4. 過去の実績を参照する
5. 見積もり根拠を残す
```

### 見積もりに含めるもの

```
✅ 含める
- 設計・実装
- テスト作成
- コードレビュー対応
- ドキュメント更新
- デプロイ・確認
- バッファ（20-50%）

❌ 忘れがち（別途計上）
- 環境構築
- 調査・学習
- MTG・コミュニケーション
- レビュー実施（他人の）
- 障害対応
```

## 2. 見積もり手法

### Tシャツサイズ

```
| サイズ | 目安時間 | 例 |
|--------|---------|-----|
| XS | 〜2h | バグ修正、文言変更 |
| S | 2-4h | 小機能、APIエンドポイント1つ |
| M | 4-8h（1日） | 中機能、画面1つ |
| L | 2-3日 | 大機能、複数画面 |
| XL | 1週間 | 新モジュール |
| XXL | 2週間以上 | → 分割必須 |
```

### 三点見積もり

```
期待値 = (楽観値 + 4×最可能値 + 悲観値) / 6

例: API認証機能
- 楽観値: 4h（スムーズにいけば）
- 最可能値: 8h（通常ペース）
- 悲観値: 16h（問題発生時）

期待値 = (4 + 4×8 + 16) / 6 = 8.7h ≈ 9h
```

### フィボナッチ（ストーリーポイント）

```
1, 2, 3, 5, 8, 13, 21, 34...

| ポイント | 相対的な大きさ |
|---------|---------------|
| 1 | 基準タスク（最小単位） |
| 2 | 基準の2倍 |
| 3 | 基準の3倍 |
| 5 | 半日〜1日 |
| 8 | 1-2日 |
| 13 | 2-3日 |
| 21+ | 分割必須 |
```

## 3. タスク分解

### 分解の粒度

```
✅ 良い粒度
- 1タスク = 2h〜1日
- 成果物が明確
- 完了条件が明確
- 依存関係が少ない

❌ 悪い粒度
- 1タスク = 3日以上（分割する）
- 「〜を検討」（成果物不明）
- 「〜の対応」（範囲不明）
```

### WBS（Work Breakdown Structure）

```markdown
## 機能A: ユーザー認証
### A-1: 設計（8h）
- A-1-1: API設計書作成（2h）
- A-1-2: DB設計書作成（2h）
- A-1-3: 設計レビュー（2h）
- A-1-4: 設計修正（2h）

### A-2: バックエンド（16h）
- A-2-1: ログインAPI（4h）
- A-2-2: ログアウトAPI（2h）
- A-2-3: トークン更新API（4h）
- A-2-4: テスト作成（4h）
- A-2-5: レビュー対応（2h）

### A-3: フロントエンド（12h）
- A-3-1: ログイン画面（4h）
- A-3-2: 認証状態管理（4h）
- A-3-3: テスト作成（2h）
- A-3-4: レビュー対応（2h）

合計: 36h = 4.5人日
バッファ30%: +1.5人日
見積もり: 6人日
```

## 4. 不確実性への対応

### コンティンジェンシー（リスクバッファ）

```
| 不確実性レベル | バッファ | 条件 |
|---------------|---------|------|
| 低 | +20% | 過去に同様の実績あり |
| 中 | +30-50% | 部分的に新しい要素 |
| 高 | +50-100% | 未経験の技術・領域 |
| 非常に高 | +100%以上 | 要件自体が不明確 |
```

### リスク係数

```
技術リスク:
- 使用経験あり: ×1.0
- 使用経験少ない: ×1.3
- 未経験: ×1.5-2.0

要件リスク:
- 要件確定: ×1.0
- 要件曖昧: ×1.3
- 要件未確定: ×1.5-2.0

外部依存リスク:
- 依存なし: ×1.0
- 外部API依存: ×1.2
- 外部チーム依存: ×1.3
```

## 5. 見積もり精度向上

### 振り返り

```markdown
## 見積もり振り返り

| タスク | 見積もり | 実績 | 差異 | 理由 |
|--------|---------|------|------|------|
| API実装 | 8h | 12h | +4h | 仕様変更 |
| テスト | 4h | 3h | -1h | 想定より簡単 |
| レビュー | 2h | 4h | +2h | 指摘多数 |

### 学び
- 仕様確定前の見積もりは+50%必要
- レビュー対応は見積もり通りにならない
```

### ベロシティ測定

```
スプリント1: 30ポイント完了
スプリント2: 28ポイント完了
スプリント3: 32ポイント完了

平均ベロシティ: 30ポイント/スプリント

→ 次スプリントは30ポイントを計画上限とする
```

## 6. スコープ調整

### MoSCoW法

```
Must（必須）: これがないとリリースできない
Should（重要）: あるべきだが、なくてもリリース可能
Could（あれば良い）: 時間があれば対応
Won't（今回はやらない）: スコープ外を明示

例:
Must:   ログイン、ユーザー登録、基本機能
Should: パスワードリセット、プロフィール編集
Could:  SNSログイン、ダークモード
Won't:  多言語対応、モバイルアプリ
```

### トレードオフ調整

```
スコープ・品質・期間・コストの4要素

期間固定の場合:
→ スコープを削る（Could → Won't）
→ 品質は下げない

スコープ固定の場合:
→ 期間を延ばす
→ リソース追加（コスト増）
```

### スコープクリープ防止

```
追加要望への対応:
1. 影響を見積もる（工数、期間、リスク）
2. トレードオフを提示（何を削るか）
3. 決定者の承認を得る
4. 変更を文書化

「追加しますか？」ではなく
「追加すると〇〇が△△になりますが、どうしますか？」
```

## 7. 見積もりテンプレート

### プロジェクト見積もり

```markdown
# {{プロジェクト名}} 見積もり

## 概要
- 期間: 〇〇〜〇〇（X週間）
- 規模: 約XX人日

## 前提条件
- 要件は確定済み
- 外部API仕様は提供済み
- レビュアーは別途確保

## 見積もり内訳

| フェーズ | 工数 | 備考 |
|---------|------|------|
| 設計 | 5人日 | 基本設計、詳細設計 |
| 開発 | 15人日 | バックエンド、フロントエンド |
| テスト | 5人日 | 単体、結合、E2E |
| バッファ | 5人日 | 20% |
| **合計** | **30人日** | |

## 機能別内訳

| 機能 | 工数 | 優先度 |
|------|------|--------|
| 認証 | 6人日 | Must |
| ダッシュボード | 8人日 | Must |
| レポート | 4人日 | Should |
| 設定 | 2人日 | Should |
| バッファ | 5人日 | - |

## リスク

| リスク | 影響 | 対策 | 追加工数 |
|--------|------|------|---------|
| 外部API遅延 | 中 | モック対応 | +2人日 |
| 要件変更 | 高 | 変更管理 | +3人日 |

## 除外事項
- 〇〇機能は対象外
- 〇〇対応は次フェーズ
```

```

```

