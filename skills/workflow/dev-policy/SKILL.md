# 開発方針スキル

## 適用タイミング

このスキルは以下の場合に読み込む：
- プロジェクト開始時
- 開発判断に迷った時
- コードレビュー時

---

## 1. 基本方針：完成より精度

### 考え方

```
❌ 早く完成させる → 後で直す → 負債が溜まる → 破綻
✅ 正しく作る → 少しずつ → 品質維持 → 長期運用可能
```

### 優先順位

```
1. 正しく動く（バグがない）
2. 理解できる（可読性）
3. 変更できる（保守性）
4. 速い（パフォーマンス）
5. 機能が多い（スコープ）

※ 5を優先して1-4を犠牲にしない
```

### 「完成」の定義

```
コードを書いた ≠ 完成

完成とは:
[ ] コードが動く
[ ] テストがある
[ ] レビューを通過
[ ] ドキュメント更新
[ ] デプロイ可能
```

## 2. 品質基準

### コード品質

| 項目 | 基準 |
|------|------|
| テストカバレッジ | 80%以上（重要パスは100%） |
| 型カバレッジ | 100%（any禁止） |
| Lint | エラー0 |
| 複雑度 | 関数あたり10以下 |
| 関数行数 | 20行以下 |
| 重複コード | 3回出たら共通化 |

### レビュー品質

```
レビューで確認すること:
[ ] 仕様を満たしているか
[ ] バグの可能性はないか
[ ] セキュリティリスクはないか
[ ] テストは十分か
[ ] 可読性は高いか
[ ] 既存コードとの整合性
```

### 納品品質

```
リリース前チェック:
[ ] 全テスト通過
[ ] ステージング環境で動作確認
[ ] セキュリティスキャン通過
[ ] パフォーマンステスト通過
[ ] ロールバック手順確認
```

## 3. 開発ルール

### コーディング

```
1. 型を書く
   - TypeScript: strictモード
   - Python: type hints + mypy

2. テストを書く
   - 実装前にテストケース洗い出し
   - 最低限、正常系・異常系・境界値

3. 小さく作る
   - 1PR = 1機能
   - 300行以下/PR
   - レビューしやすいサイズ

4. 命名に時間をかける
   - 読んで意味がわかる名前
   - 略語は避ける
   - 検索しやすい名前
```

### Git運用

```
1. mainは常にデプロイ可能
2. feature/*で作業
3. PRは1日以内にレビュー
4. Squash mergeでコミット整理
5. コミットメッセージは具体的に
```

### タスク管理

```
1. 見積もりは1.5倍で計算
2. 3日以上かかるタスクは分割
3. 不明点は着手前に解消
4. ブロッカーは即共有
5. 完了条件を明確に
```

## 4. 技術的負債への対応

### 負債を作らない

```
✅ 最初から正しく作る
✅ 「後で直す」は許可制
✅ TODOにはチケット番号必須
✅ ハックには理由コメント必須
```

### 負債の可視化

```
// TODO(#123): リファクタリング - 2025-03-01まで
// HACK: 外部APIの仕様バグ回避。API v2で解消予定
// FIXME(#456): パフォーマンス問題あり
```

### 負債の返済

```
毎スプリント20%を改善に充てる:
- リファクタリング
- テスト追加
- ドキュメント整備
- 依存関係更新
```

## 5. 判断基準

### 実装方法に迷ったら

```
問い:
1. 3ヶ月後の自分が理解できるか？
2. 他のメンバーがメンテできるか？
3. 要件変更に対応しやすいか？

→ 全部Yesなら採用
→ 1つでもNoなら再検討
```

### スコープに迷ったら

```
MVP優先:
1. なくても動くか？ → 後回し
2. 代替手段があるか？ → 後回し
3. ユーザーの主要タスクに必須か？ → 今やる
```

### 速度と品質に迷ったら

```
原則: 品質を落とさない

例外（承認必須）:
- ビジネスクリティカルな期限
- 明確な返済計画がある
- 影響範囲が限定的
```

## 6. アンチパターン

### やってはいけないこと

```
❌ 「動いたからOK」でPR
❌ テストなしでマージ
❌ レビューを急かす
❌ 「時間がないから」でハック
❌ ドキュメントを後回し
❌ エラーを握りつぶす
❌ console.log残し
❌ コメントアウト残し
```

### 危険なサイン

```
🚨 「とりあえず」が増えている
🚨 バグが減らない
🚨 新機能追加が遅くなってきた
🚨 特定の人しか触れないコードがある
🚨 「昔からこうだから」で思考停止
```

## 7. チーム文化

### コードレビュー文化

```
レビュアー:
- 人格ではなくコードを見る
- 「なぜ」を説明する
- 良い点も伝える
- 質問形式を使う（「〜した理由は？」）

レビュイー:
- 指摘は学びの機会
- 防御的にならない
- 不明点は質問する
- 感謝を伝える
```

### 心理的安全性

```
✅ 質問を歓迎する
✅ 失敗を責めない（仕組みを改善）
✅ 「わからない」と言える
✅ 反対意見を尊重する
```

### 継続的改善

```
定期的に振り返る:
- 何がうまくいったか
- 何が問題だったか
- 次に何を試すか

小さく試す:
- 1スプリントで1つ改善
- 効果を測定
- 良ければ継続、悪ければ戻す
```

## チェックリスト

### コード作成時

```
[ ] 型定義が完全か
[ ] テストが書かれているか
[ ] エラーハンドリングは適切か
[ ] ログは適切に出力されるか
[ ] セキュリティ考慮しているか
```

### PR作成時

```
[ ] 1PR1機能になっているか
[ ] テストが通っているか
[ ] セルフレビューしたか
[ ] 説明文は十分か
[ ] 関連チケットをリンクしたか
```

### マージ前

```
[ ] レビュー承認されたか
[ ] CIが通っているか
[ ] コンフリクトは解消したか
[ ] ドキュメントは更新したか
```
