# 設計書スキル

## 適用タイミング

このスキルは以下の場合に読み込む：
- 新規プロジェクト開始時
- 企画書から設計書を作成する時
- 設計レビュー時

---

## 1. 設計フロー

```
企画書（Why/What）
    ↓
基本設計（What/How概要）
    ↓
詳細設計（How詳細）
    ↓
実装
```

### 各フェーズの成果物

| フェーズ | 成果物 | 粒度 |
|---------|--------|------|
| 企画 | 企画書、要件定義 | なぜ作るか、何を作るか |
| 基本設計 | システム構成図、画面一覧、API一覧、ER図 | 全体像、主要機能 |
| 詳細設計 | API仕様、テーブル定義、画面仕様、処理フロー | 実装可能な粒度 |

---

## 2. 企画書から基本設計

### 企画書の読み方

```
抽出するもの：
□ 目的・背景（なぜ作るか）
□ ターゲットユーザー（誰のため）
□ 主要機能（何ができる）
□ 成功指標（どうなれば成功）
□ 制約条件（期間、予算、技術）

## 3. 基本設計から詳細設計

### 詳細設計の粒度

```
基本設計: 「ユーザー登録機能を作る」
詳細設計: 「POST /api/users にname, email, passwordを送ると
          バリデーション後、bcryptでハッシュ化してDBに保存、
          JWTを返す」
```

### 詳細設計書テンプレート

```markdown
# 詳細設計書

## 1. API仕様

### API-001: ユーザー登録
#### エンドポイント
POST /api/v1/users

#### リクエスト
```json
{
  "name": "string (1-100文字)",
  "email": "string (メール形式)",
  "password": "string (8文字以上)"
}
```

#### レスポンス（成功）
```json
{
  "success": true,
  "data": {
    "id": "string",
    "name": "string",
    "email": "string",
    "createdAt": "datetime"
  }
}
```

#### レスポンス（エラー）
| コード | 条件 | メッセージ |
|--------|------|-----------|
| 400 | バリデーション | 各フィールドのエラー |
| 409 | メール重複 | Email already exists |

#### 処理フロー
1. リクエストバリデーション
2. メール重複チェック
3. パスワードハッシュ化（bcrypt cost=12）
4. DBにINSERT
5. レスポンス生成

## 2. テーブル定義

### users
| カラム | 型 | NULL | デフォルト | 説明 |
|--------|-----|------|-----------|------|
| id | SERIAL | NO | - | 主キー |
| name | VARCHAR(100) | NO | - | 名前 |
| email | VARCHAR(255) | NO | - | メール（UNIQUE） |
| password_hash | VARCHAR(255) | NO | - | パスワードハッシュ |
| created_at | TIMESTAMPTZ | NO | NOW() | 作成日時 |
| updated_at | TIMESTAMPTZ | NO | NOW() | 更新日時 |

#### インデックス
- PRIMARY KEY (id)
- UNIQUE (email)

#### 制約
- name: 1-100文字
- email: メール形式、UNIQUE

## 4. 設計レビュー観点

### 基本設計レビュー

```
□ 要件との整合性
  - 企画書の要件をすべてカバーしているか
  - スコープ外が明確か

□ 技術選定の妥当性
  - 選定理由が明確か
  - チームのスキルとマッチするか
  - 将来の拡張性

□ 実現可能性
  - 期間内に実装可能か
  - 技術的に実現可能か
  - リスクは洗い出されているか
```

### 詳細設計レビュー

```
□ 実装可能な粒度か
  - エンジニアが迷わず実装できるか
  - 曖昧な部分はないか

□ 整合性
  - API仕様とテーブル定義は整合しているか
  - 画面仕様とAPI仕様は整合しているか

□ エラーケース
  - 異常系が網羅されているか
  - エラーメッセージは適切か

□ 非機能要件
  - 性能要件を満たせる設計か
  - セキュリティ考慮されているか
```

## 5. 設計Tips

### 良い設計書の特徴

```
✅ 読み手が実装できる粒度
✅ 図と文章のバランス
✅ 変更履歴が追える
✅ 用語が統一されている
✅ 曖昧な表現がない
```

### 避けるべきこと

```
❌ 実装コードをそのまま書く
❌ 詳細すぎて変更に弱い
❌ 抽象的すぎて実装できない
❌ 誰も読まない長大なドキュメント
```

## チェックリスト

### 基本設計完了時

```
[ ] 全機能が一覧化されている
[ ] 主要画面が一覧化されている
[ ] 主要APIが一覧化されている
[ ] ER図がある
[ ] 非機能要件が定義されている
[ ] 技術スタックが決定している
```

### 詳細設計完了時

```
[ ] 全APIの仕様が定義されている
[ ] 全テーブルの定義がある
[ ] 全画面の仕様がある
[ ] 処理フローが明確
[ ] エラーケースが網羅されている
[ ] レビュー完了している
```
