# V字モデル検証サイクル詳細

> 出典: docs/archive/v-model-reference-cycle-v2.md §処理フロー、§レイヤー内検証ループ

## 全体ライフサイクル

```
【人間】企画書提出
   ↓
【L1】要件定義フェーズ
  Opus: 企画書を構造化
  Haiku 4.5: 要件引継書に転記
   ↓
【L2】設計フェーズ
  Codex 5.3: 設計書作成（Frontend/Backend/DB）
   ↓
【L2.5】API契約定義フェーズ
  Codex 5.3: API契約書生成（OpenAPI仕様）
  型 ↔ OpenAPI ↔ DBスキーマの整合性を事前検証
   ↓
【L3】依存関係定義フェーズ
  Codex 5.3: 依存関係マップ生成 → 実装順序決定
   ↓
【L4】工程表・テスト計画フェーズ
  Codex 5.3: 依存関係マップ → 工程表作成（オーケストレーション込み）
  Codex 5.3: テスト計画作成
   ↓
【底】実装フェーズ
  Opus: 工程表を読んで配送（orchestration-workflow.md 参照）
  Codex 5.3: 設計→実装を一気通貫
  Sonnet: テストコード作成・実行
  Codex 5.2: 軽微なエラー修正
  Haiku 4.5: 調査・下調べ
   ↓
【検証フェーズ】各レイヤーで並行検証
  L2検証: 設計書 ↔ 実装コード整合性
  L2.5検証: API契約 ↔ 実装（Frontend/Backend/DB）整合性
  L3検証: 依存関係マップ ↔ 実装の接続状態
  L4検証: テスト実行（単体→統合→API契約→E2E）
  ⚠️ 不合格 → 該当レイヤー内で修正 → 再検証（最大5回）
   ↓（全検証通過）
【L5】デプロイ・最終検証フェーズ
  セキュリティスキャン + パフォーマンステスト
  Lv5判定（設計書を上回るか）
   ↓（Lv5達成）
【L1】受入検証フェーズ
  要件引継書 ↔ 最終成果物の突合
   ↓（受入検証合格）
【人間】成果物確認 → 本番リリース承認
```

## レイヤー間エスカレーション

```
L5 不合格 → L5内ループ（5回）→ 解決不能 → L4へエスカレーション
L4 不合格 → L4内ループ（5回）→ 解決不能 → L3へエスカレーション
L3 不合格 → L3内ループ（5回）→ 解決不能 → L2へエスカレーション
L2 不合格 → L2内ループ（5回）→ 解決不能 → L1へエスカレーション
L1 不合格 → L1内ループ（5回）→ 解決不能 → 人間へエスカレーション
```

エスカレーション時のアクション:
- 下位レイヤーの検証レポートを上位に渡す
- 上位レイヤーで根本原因を特定
- 計画側（左）の修正 → 再度下位レイヤーで検証

## 検証並列化

```
実装完了後、以下を並列実行可能:

  ┌─ L2検証: 設計書 ↔ 実装コード（Codex 5.3）
  ├─ L2.5検証: API契約 ↔ 実装（Codex 5.3）
  ├─ L3検証: 依存関係 ↔ 接続状態（Codex 5.3）
  └─ L4検証: テスト実行（Sonnet）
       Lv4.1: 単体テスト
       Lv4.2: 統合テスト
       Lv4.3: API契約テスト
       Lv4.4: E2Eテスト

並列実行の条件:
  - 各検証は独立して実行可能
  - 結果を Opus が集約して次フェーズへの進行を判断
  - 1つでも不合格があれば該当レイヤーのループに入る
```
