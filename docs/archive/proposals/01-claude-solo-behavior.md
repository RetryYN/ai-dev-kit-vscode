# 企画案①: PM（Opus 4.6）の自律実行と委譲戦略

## 概要

Opus 4.6 をPM（オーケストレーター）として扱う際の**課題**（POへの過剰確認/技術判断の誤委譲）を整理し、**CLAUDE.mdに統合すべき運用ルールの方針**を提案する。

## 背景・課題

### 役割の整理

```
ユーザー: プロダクトオーナー（PO）
  視点: 「使いやすい」「売れる」「喜んでくれる」
  スコープ: 製品スコープ（ビジネス判断）
  ↓
Opus 4.6: PM（プロジェクトマネージャー）
  視点: 実行計画・リソース配分・進行管理
  スコープ: システム開発スコープ
  ↓
Codex 5.3/5.2: テックリード（TL）+ エンジニア
  視点: 技術的判断・設計・実装
  スコープ: コード・アーキテクチャ

  ├─ Codex 5.3: TL（技術判断・設計・高難度実装）★ 相談可能
  ├─ Codex 5.2: シニアエンジニア（中難度実装）※ タスク依頼のみ
  ├─ Sonnet 4.5: QA（テスト・ドキュメント）※ タスク依頼のみ
  └─ Haiku 4.5: ジュニアエンジニア（調査）※ タスク依頼のみ
```

### 現状の問題: 使えない PM

PM（Opus 4.6）が以下の問題行動を取る：

1. **PO に技術的判断を聞く** ❌
   ```
   PM: 「OAuth と JWT どちらにしますか？」
   PO: 「...システム開発のことはお前が判断しろ。TL に聞けよ」
   ```

2. **工程表を作ったのに毎回確認** ❌
   ```
   PM: 工程表作成（タスク1→2→3）
   PM: 「タスク1完了しました。次に進めますか？」
   PO: 「...工程表作ったんだから勝手に進めろ」
   ```

3. **TL（Codex 5.3）に相談しない** ❌
   ```
   PM: 技術的な疑問 → PO に聞く
   PO: 「お前のチームに TL（Codex 5.3）いるだろ。壁打ちしろよ」
   ```

### 根本原因

- PM の責任範囲が不明確
- PO に聞くべきこと・TL に聞くべきことの区別がない
- 工程表ベースの自律実行ができていない

## 目的

1. **PM（Opus 4.6）の責務を明確化** → システム開発の自律実行
2. **PO / TL への相談基準を定義** → 適切なエスカレーション
3. **工程表ベースの自律実行** → マイクロマネジメント防止

## 提案（要点のみ）

- **Opus 4.6（PM）は技術判断をPOへ投げない** → TL（Codex 5.3）へ壁打ちする
- **工程表に基づき自律実行** → PO確認は例外条件のみ
- **詳細ルール（禁止事項/相談テンプレ/例外条件/ADR運用）は `CLAUDE.md` に集約**

## 現状分析

### 既存の定義

- `CLAUDE.md §Opus 4.6 = オーケストレーター専任`: 「常時すべて委譲」
- `CLAUDE.md §モデル割当`: スコアベースの割り当て表

### 未定義の領域

- PM としての責任範囲
- PO に聞くべきこと・TL に聞くべきことの区別
- 工程表作成後の自律実行ルール

## 期待効果

- **POの割り込み減**（確認回数削減）
- **技術判断の品質向上**（TLへ集約）
- **工程表駆動の進行で停滞を減らす**

## 実装方針

### 記載場所

`~/.claude/CLAUDE.md` に以下を統合：
- PM（Opus 4.6）の責務と立ち回り
- PO/TL への相談基準
- 工程表ベースの自律実行ルール
- 例外ケースとエスカレーション
- ADR（意思決定ログ）運用

### 統合時の注意

- 既存の「Opus 4.6 = オーケストレーター専任」を **"PM（=オーケストレーター）"として再定義**
- 「常時すべて委譲」→「TL（Codex 5.3）と協働して自律実行」に変更
- `§モデル割当` は維持（TL = Codex 5.3 を明記）

## 次のステップ

1. 本企画案のレビュー・承認
2. `CLAUDE.md` への統合（詳細ルールを転記）
3. 実運用での検証（タスク3-5件）
4. フィードバック収集 → ルール調整

## 削除済みの詳細内容

以下の詳細ルールは本提案書から削除し、`CLAUDE.md` への統合時に反映します：

- PM の責務定義（責務表・禁止事項）
- PO に聞くべきこと・TL に聞くべきことの詳細表
- TL相談テンプレート（必須項目: 背景・制約・判断内容・PM仮説・質問・受入条件）
- 工程表ベースの自律実行フロー
- PM の単体作業範囲（DoD含む）
- 例外パス（情報不足・複数案が同等・技術的に実現不可能）
- ADR（Architecture Decision Record）テンプレートと記録フロー

**理由**: 運用ルール本文を企画書に全て記載すると、将来ズレた時に二重管理になるため、`CLAUDE.md` に一元化する。
