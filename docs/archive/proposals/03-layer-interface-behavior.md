# 企画案③: 階層間連携の立ち回り定義

## 概要

HELIX フレームワークの L1〜L6 + 受入 の遷移条件・成果物フォーマット・逆流フィードバックルールを明確に定義し、PM（Opus 4.6）が TL（Codex 5.3）と協働して階層間のゲートを適切に管理する。

## 背景・課題

### 役割の再確認

```
PO（ユーザー）: 製品スコープ
  「これが欲しい」「これが売れる」
  ↓
PM（Opus 4.6）: システム開発の実行管理
  階層間の遷移判断・ゲート管理
  ↓
TL（Codex 5.3）: 技術判断 + 実装
  各階層の技術的検証・成果物作成
```

### 現状の問題

HELIX は階層開発モデルだが、**階層間のインターフェースが曖昧**：

```
L1 要件 ─┐
L2 設計 ─┤
L2.5 API ─┤ ← この「階層間」の受け渡しルールが未定義
L3 依存 ─┤
L4 工程 ─┤
L5 デプロイ─┤
受入 ─────┘
```

### 具体的な問題

| 問題 | 現状 | 影響 |
|------|------|------|
| **一方通行** | L2完了 → L2.5開始（自動遷移） | L2.5で問題発見してもL2に戻せない |
| **成果物が曖昧** | 「設計書」と言っているが何を含むべきか不明 | L2.5が必要な情報を受け取れない |
| **検証基準なし** | L2完了の判定基準がない | 不完全なまま次の階層に進む |
| **逆流ルール不在** | 下位層で問題発見時の処理が未定義 | PM が手動で戻すしかない |

### 具体例

```
PM: L2 設計フェーズ完了
  → TL（Codex 5.3）に「ユーザー認証API」の設計書作成依頼
  ↓
PM: L2.5 API契約フェーズ開始
  → TL に OpenAPI 仕様生成依頼
  ↓
TL: エラー「認証方式が設計書に記載されていない」

現状: PM が手動で L2 に戻って修正
理想: 自動的に L2 に Issue 返却 → TL が修正 → L2.5 再検証
```

## 目的

1. **階層間コントラクトの定義** → 各階層の入出力を明確化
2. **階層間ゲートの設置** → 遷移条件を満たさない限り次に進まない
3. **逆流フィードバックループの確立** → 下位層の問題を上位層で修正
4. **PM と TL の役割分担** → ゲート判断は PM、技術検証は TL

## 現状分析

### 既存の定義

- `SKILL_MAP.md`: 階層の一覧と担当スキルを定義
- `orchestration-workflow.md`: フェーズ別の I/O サマリーを記載

```markdown
| フェーズ | 入力 | 出力 | スキル |
| L2 設計 | 要件リスト | 設計書（画面/API/DB） | design-doc, api, db, security |
| L2.5 API契約 | 設計書 | OpenAPI仕様 + 型定義 | api-contract |
```

### 未定義の領域

- 「設計書」の具体的なフォーマット・必須項目
- L2 → L2.5 の遷移条件（何を満たせば次に進めるか）
- L2.5 で問題発見時の処理（どうやって L2 に戻すか）
- PM と TL の役割分担（誰が何を判断するか）

## 提案内容

### 用語の前提（SKILL_MAP準拠）

- **フェーズ**: `L1 / L2 / L2.5 / L3 / L4 / 実装 / 検証 / L5（デプロイ）/ 受入`
- **レイヤー表記とフェーズ表記を混同しない**（本提案は"階層間I/O"だが、運用フローはSKILL_MAPのフェーズ名を採用）
- **検証フェーズは独立**（L2-L5各層の検証を並列で実施）

### ③-1. 階層間コントラクトの定義

各階層間の入出力・検証内容・ゲート条件・失敗時処理を明確化：

#### L1 → L2（要件 → 設計）

| 項目 | 内容 | PM の役割 | TL の役割 |
|------|------|-----------|-----------|
| **入力要件** | 要件リスト + 仮定一覧 | 要件を TL に伝達 | - |
| **出力成果物** | 設計書（画面・API・DB・セキュリティ） | 成果物の完成度確認 | 設計書作成 |
| **検証内容** | 全要件が設計書にトレース可能か | - | 要件カバレッジ検証 |
| **ゲート条件** | ① ユーザー要望充足<br>② 技術的課題クリア<br>③ 下方スコープなし | ゲート通過判定 + スコープ管理 | 技術的検証 |
| **失敗時処理** | L1 に差し戻し（要件の明確化） | PO に要件確認 | - |

#### L2 → L2.5（設計 → API契約）

| 項目 | 内容 | PM の役割 | TL の役割 |
|------|------|-----------|-----------|
| **入力要件** | 設計書（API一覧・型定義・認証方式・エラーハンドリング） | 設計書を TL に伝達 | - |
| **出力成果物** | OpenAPI 仕様 + TypeScript 型定義 | 成果物の完成度確認 | OpenAPI 仕様作成 |
| **検証内容** | APIエンドポイント・型・認証方式の整合性 | - | api-contract スキルで検証 |
| **ゲート条件** | ① ユーザー要望充足<br>② 技術的課題クリア<br>③ 下方スコープなし | ゲート通過判定 + スコープ管理 | 技術的検証 |
| **失敗時処理** | L2 に Issue 返却 + 修正依頼 | L2 修正指示 | 不足情報・不整合を指摘 |

#### L2.5 → L3（API契約 → 依存関係）

| 項目 | 内容 | PM の役割 | TL の役割 |
|------|------|-----------|-----------|
| **入力要件** | OpenAPI 仕様 + 型定義 + DB設計 | 成果物を TL に伝達 | - |
| **出力成果物** | 依存マップ + 実装順序 | 実装順序の承認 | 依存関係分析 |
| **検証内容** | 循環依存の検出・実装可能性検証 | - | dependency-map スキルで検証 |
| **ゲート条件** | ① ユーザー要望充足<br>② 技術的課題クリア<br>③ 下方スコープなし | ゲート通過判定 + スコープ管理 | 技術的検証 |
| **失敗時処理** | L2.5 に差し戻し（API設計見直し） | L2.5 修正指示 | API設計の問題点を指摘 |

#### L3 → L4（依存関係 → 工程表）

| 項目 | 内容 | PM の役割 | TL の役割 |
|------|------|-----------|-----------|
| **入力要件** | 依存マップ + 実装順序 | 依存マップを受け取り | - |
| **出力成果物** | 工程表（タスクID・難易度・担当・スキル） | **工程表作成（PM の主要業務）** | 技術的難易度の評価 |
| **検証内容** | 全タスクに割り当て完了・並列実行可能性 | 工程表の完全性確認 | 技術的実現可能性検証 |
| **ゲート条件** | ① ユーザー要望充足<br>② 技術的課題クリア<br>③ 下方スコープなし | ゲート通過判定 + スコープ管理 | 技術的検証 |
| **失敗時処理** | L3 に差し戻し（実装順序再検討） | L3 修正指示 | 実装順序の問題点を指摘 |

#### L4 → 実装（工程表 → 実装）

| 項目 | 内容 | PM の役割 | TL の役割 |
|------|------|-----------|-----------|
| **入力要件** | 工程表 + 設計書 + API仕様 | **工程表に従って自律実行** | - |
| **出力成果物** | コード + テスト + ドキュメント | 進行管理 | 実装（企画案②の5階層） |
| **検証内容** | 全工程表タスクが完了しているか | 全タスク完了確認 | codex review 合格 |
| **ゲート条件** | ① ユーザー要望充足<br>② 技術的課題クリア<br>③ 下方スコープなし | ゲート通過判定 + スコープ管理 | 技術的検証 |
| **失敗時処理** | 該当タスク再実行（最大3回）→ L4 差し戻し | 工程表見直し | 実装の問題点を指摘 |

#### 実装 → 検証（実装 → テスト検証）

| 項目 | 内容 | PM の役割 | TL の役割 |
|------|------|-----------|-----------|
| **入力要件** | 実装コード + テスト | 成果物を TL に伝達 | - |
| **出力成果物** | 検証レポート（L2-L5 各層の検証結果） | 検証レポート確認 | 各層の技術的検証 |
| **検証内容** | 設計整合性・API契約遵守・カバレッジ・品質 | - | quality-lv5 スキルで検証 |
| **ゲート条件** | ① ユーザー要望充足<br>② 技術的課題クリア<br>③ 下方スコープなし | ゲート通過判定 + スコープ管理 | 技術的検証 |
| **失敗時処理** | レイヤー内ループ（最大5回）→ 上位層エスカレ | エスカレーション判断 | 問題点の特定 |

#### 検証 → L5（検証 → デプロイ）

| 項目 | 内容 | PM の役割 | TL の役割 |
|------|------|-----------|-----------|
| **入力要件** | 検証済みコード + 検証レポート | 成果物を TL に伝達 | - |
| **出力成果物** | デプロイ結果 + パフォーマンステスト結果 | デプロイ承認 | デプロイ実行 |
| **検証内容** | セキュリティスキャン・パフォーマンス基準 | - | セキュリティ・パフォーマンス検証 |
| **ゲート条件** | ① ユーザー要望充足<br>② 技術的課題クリア<br>③ 下方スコープなし | ゲート通過判定 + スコープ管理 | 技術的検証 |
| **失敗時処理** | L5 に差し戻し（修正） | L5 修正指示 | 問題点の特定 |

#### L5 → 受入（デプロイ → 受入）

| 項目 | 内容 | PM の役割 | TL の役割 |
|------|------|-----------|-----------|
| **入力要件** | 最終成果物 + 要件リスト（L1） | 受入判定準備 | - |
| **出力成果物** | 受入判定レポート | **PO に報告** | 技術的な受入検証（verification §14 の観点で支援） |
| **検証内容** | 要件リスト ↔ 最終成果物の突合 | - | verification スキルで検証 |
| **ゲート条件** | ① ユーザー要望充足<br>② 技術的課題クリア<br>③ 下方スコープなし | **PO に承認依頼** + スコープ管理 | 技術的検証 |
| **失敗時処理** | 該当レイヤーに差し戻し | エスカレーション | 要件未達の特定 |
| **最終承認** | **PO（人間）が最終承認** | 受入判定材料を提出し承認依頼 | - |

### ③-2. 階層間ゲートの実装方法

各階層完了時に**PM が TL に検証を依頼 + スコープ管理**：

```bash
# 例: L2 完了時
PM: "L2 設計完了。L2.5 に進む前にゲート検証を実施"

# [PM → TL にゲート検証依頼]
PM: verification スキルを起動（L2 層検証）
TL: 技術的検証実行
  - 要件カバレッジ
  - API一覧の完全性
  - 認証方式の記載

# [TL から PM に検証結果]
TL: 検証結果を報告

# [PM がゲート判定: 3つの観点]
PM: ゲート判定
  ① ユーザー要望充足: PO の要求が全て設計書に含まれているか
  ② 技術的課題クリア: TL の検証が全て合格
  ③ 下方スコープなし: 勝手に機能を削ったり簡略化していないか

if ①②③ 全て OK; then
  PM: "L2.5 開始"
else
  PM: "L2 に差し戻し"
  PM → TL: Issue 作成
    「不足項目: 認証方式が設計書に記載されていない」
    「修正依頼: OAuth 2.0 / JWT のどちらを使うか設計書に明記してください」
  TL: 修正実施
  PM: ゲート再検証（合格まで）
fi
```

### ③-3. 逆流フィードバックループ

下位層で問題発見時の処理フロー：

```
L2.5 で TL が問題検出（例: 型定義に不整合）
  ↓
TL → PM に報告: 「User 型の email フィールドが設計書にないが API仕様にある」
  ↓
PM: L2 に Issue 作成（TL に修正依頼）
  ↓
TL: L2 修正実施
  ↓
PM: L2 ゲート再検証（TL に依頼）
  ↓ 合格
PM: "L2.5 再開"
  ↓
TL: L2.5 作業再開（修正された設計書を使用）
  ↓
PM: L2.5 ゲート検証（TL に依頼）
  ↓ 合格
PM: "L3 開始"
```

### ③-4. PM と TL の役割分担まとめ

| 役割 | PM（Opus 4.6） | TL（Codex 5.3） |
|------|-----------|-------------|
| **階層間遷移判断** | ✅ ゲート通過判定（3観点） | - |
| **スコープ管理** | ✅ 下方スコープ防止 | - |
| **技術的検証** | - | ✅ 各層の検証実行 |
| **成果物作成** | 工程表作成（L4） | 設計・実装・検証 |
| **問題発見時** | ✅ 差し戻し判断 | ✅ 問題の特定・報告 |
| **PO への報告** | ✅ 受入判定・完了報告 | - |

**PM はマネジメント + スコープ管理、TL は技術実務**

**ゲート判定の3観点:**
1. **ユーザー要望充足**: PO の要求が満たされているか
2. **技術的課題クリア**: TL の技術検証が合格
3. **下方スコープなし**: 勝手に機能を削ったり簡略化していないか

### ③-5. ゲート判定チェックリスト

PM が各階層完了時に確認する項目：

#### L2 設計 → L2.5 API契約

```markdown
## L2 → L2.5 ゲート判定

### ① ユーザー要望充足
- [ ] 全要件が設計書に記載されているか
- [ ] PO が求める機能が全て含まれているか
- [ ] 削除・簡略化した機能はないか

### ② 技術的課題クリア（TL 検証）
- [ ] API一覧が完全か（CRUD全て定義）
- [ ] 認証方式が記載されているか
- [ ] エラーハンドリングが設計されているか
- [ ] DB設計が完了しているか

### ③ 下方スコープなし
- [ ] 要件リスト（L1）と設計書（L2）が1:1対応しているか
- [ ] 勝手に「簡易版」にしていないか
- [ ] 「後で実装」で先送りした機能はないか

判定: [合格 / 要修正]
```

#### L2.5 API契約 → L3 依存関係

```markdown
## L2.5 → L3 ゲート判定

### ① ユーザー要望充足
- [ ] 全APIが設計書のAPI一覧と一致しているか
- [ ] OpenAPI 仕様に記載漏れはないか

### ② 技術的課題クリア（TL 検証）
- [ ] 型定義が完全か（TypeScript 型 ↔ DB スキーマ整合）
- [ ] API ↔ 型 ↔ DB の3者が整合しているか
- [ ] 認証方式が実装可能か

### ③ 下方スコープなし
- [ ] 設計書（L2）のAPI一覧が全てOpenAPI化されているか
- [ ] 省略したエンドポイントはないか

判定: [合格 / 要修正]
```

### ③-8. 例外パス（5回ループ不合格時）

#### パターン1: レイヤー内で5回ループ不合格

```
L2.5 API契約フェーズ
  ↓
TL: OpenAPI 仕様生成 → PM: ゲート検証 → 不合格（1回目）
  ↓
TL: 修正 → PM: ゲート検証 → 不合格（2回目）
  ↓
（5回繰り返し）
  ↓
PM: エスカレーション判断
  ├─ 上位層に差し戻し（L2 に問題がある）
  │   ↓
  │   L2 修正 → L2.5 再開
  │
  └─ PO に報告（要件に問題がある）
      ↓
      PO: 要件明確化 → L1 から再開
```

#### パターン2: 技術的に実現不可能

```
L3 依存関係フェーズ
  ↓
TL: 依存マップ生成
  ↓
TL → PM: 「循環依存が解消できない。設計の根本的な問題」
  ↓
PM: L2 に差し戻し（設計見直し）
  ↓
TL: L2 修正（アーキテクチャ変更）
```
