# 企画案②: PM（Opus 4.6）+ TL（Codex 5.3）複合の立ち回り定義

## 概要

PM（Opus 4.6）と TL（Codex 5.3）の協働における実装フェーズの立ち回りを、**5階層のサブフェーズ**として定義し、レビュー・テスト・ドキュメントの抜け漏れを防止する。

## 背景・課題

### 役割の再確認

```
PO（ユーザー）: 製品スコープ
  ↓
PM（Opus 4.6）: システム開発の実行管理
  ↓
TL（Codex 5.3）: 技術判断 + 実装

チーム構成:
├─ Codex 5.3: TL（技術判断・設計・高難度実装）
├─ Codex 5.2: シニアエンジニア（中難度実装）
├─ Sonnet 4.5: QA（テスト・ドキュメント）
└─ Haiku 4.5: ジュニアエンジニア（調査）
```

### 現状の問題

実装フェーズで以下が頻発：

1. **レビューを抜かす**
   - PM: TL（Codex 5.3）に実装指示
   - TL: 実装完了
   - PM: 「完了！」← レビューなし ❌

2. **ドキュメント更新を忘れる**
   - コード実装だけして README/コメントを放置

3. **テストを書かない**
   - QA（Sonnet 4.5）にテスト作成を依頼し忘れる

### 具体例

```bash
# 現状の実装フロー（問題あり）
PM: codex exec "機能X実装" -m gpt-5.3-codex
TL: 実装完了
PM: 「完了！」 ← ❌

# 抜けているもの
# - codex review（TL によるレビュー）
# - レビュー指摘の修正
# - テスト追加（QA）
# - ドキュメント更新（QA）
```

### 根本原因

**実装フェーズ自体が階層構造を持っているのに、階層間ゲートが未定義**

```
実装フェーズ
├─ コード実装   ← やる
├─ レビュー     ← スキップ ❌
├─ 修正        ← スキップ ❌
├─ テスト      ← スキップ ❌
└─ ドキュメント ← スキップ ❌
```

## 目的

1. **実装フェーズを5階層のサブフェーズに分解** → 抜け漏れ防止
2. **各階層のゲート条件を定義** → 強制的に全階層を通過
3. **PM のオーケストレーション手順を明確化** → 自動化可能な形に

## 現状分析

### 既存の定義

- `ai-coding/SKILL.md §4`: マルチエージェント戦略
  - モデル別の担当は定義されている
  - しかし**実装フェーズ内の手順**は未定義

```markdown
## 4. マルチエージェント戦略（現行）

| モデル | 担当 |
| Codex 5.3 | TL: 実装（スコア8+） |
| Sonnet 4.5 | QA: テスト・ドキュメント |
```

### 問題点

- 「実装」と一言で言っているが、実際には5つのサブタスクがある
- サブタスク間の順序・ゲート条件が未定義
- PM が「実装終わり」と判断する基準が曖昧

## 提案内容

### ②-1. 実装フェーズの5階層定義

実装タスクは以下の5階層を**順番に通過**する（スキップ不可）：

| 階層 | 名称 | 担当 | 手段 | 成果物 | DoD（完了の定義） |
|------|------|------|------|--------|------------------|
| 実装.1 | コード実装 | TL (Codex 5.3 / Codex 5.2) | `codex exec` | コード | 要件を満たすコードが実装済み |
| 実装.2 | レビュー | **TL（Codex 5.3）** | `codex review --uncommitted`（**TLが実行**） | レビュー結果 | レビュー完了 + 指摘事項リスト作成 |
| 実装.3 | 修正 | Engineer (Codex 5.2) | `codex exec`（修正プロンプト） | 修正コード | レビュー指摘事項が全て解消 |
| 実装.4 | テスト | QA (Sonnet 4.5) | Task tool (general-purpose) | テストコード | テスト全て成功 + **カバレッジ基準（プロジェクト既定がある場合のみ）を満たす**／既定がない場合は `quality-lv5` の指標で代替 |
| 実装.5 | ドキュメント | QA (Sonnet 4.5) | Task tool (general-purpose) | ドキュメント更新 | 関連ドキュメント全て更新済み |

#### 実行主体（codex review）

- `codex review --uncommitted` の実行主体は **TL（Codex 5.3）** とする（PMは実行しない／結果を受けてゲート判定する）
- 例外: TL不在時のみ PMが代行し、結果をTLにエスカレーションして最終判断

#### カバレッジ基準の扱い

- **原則**: プロジェクトに既存の基準（例: coverage threshold / CIゲート）がある場合はそれに従う
- **基準がない場合**: "増分の重要ロジックにテストがあること"を必須条件にし、数値は `quality-lv5` の測定（または導入提案）に回す

### ②-2. 各階層のゲート条件

| 遷移 | ゲート条件 | 失敗時の処理 |
|------|-----------|-------------|
| 実装.1 → 実装.2 | コード実装完了 | 自動遷移（必須） |
| 実装.2 → 実装.3 | レビュー指摘事項あり | 指摘あり → 実装.3 / なし → 実装.4 |
| 実装.3 → 実装.4 | レビュー再実行で合格 | 不合格 → 実装.3 ループ（最大3回） |
| 実装.4 → 実装.5 | テスト全て成功 | 失敗 → 実装.3（コード修正） |
| 実装.5 → 完了 | 関連ドキュメント全て更新 | 自動遷移（完了判定） |

### ②-3. PM（Opus 4.6）のオーケストレーション手順

実装タスク受領時の標準フロー：

```bash
# [事前] PM が難易度スコアを計算
難易度スコア = estimation SKILL.md §9 で計算

# [実装.1] TL（Codex 5.3）に実装指示
if スコア >= 8; then
  codex exec "タスク内容" -m gpt-5.3-codex
else
  codex exec "タスク内容" -m gpt-5.2-codex
fi

# [実装.2] TL（Codex 5.3）によるレビュー（必須）
codex review --uncommitted

# [実装.3] 修正（条件付き）
if [ レビュー指摘あり ]; then
  codex exec "レビュー指摘修正: <指摘内容>" -m gpt-5.2-codex
  codex review --uncommitted  # 再検証（合格まで最大3回）
fi

# [実装.4] QA（Sonnet 4.5）にテスト作成依頼
Task tool:
  prompt: "タスク内容のテストを追加・実行。既存テストが壊れていないか確認。"
  subagent_type: general-purpose
  model: sonnet

# [実装.5] QA（Sonnet 4.5）にドキュメント更新依頼
Task tool:
  prompt: "タスク内容に関連する以下を更新: README/JSDoc/コメント/設計書"
  subagent_type: general-purpose
  model: sonnet

# [完了判定]
PM: 全階層通過 → ✅ 実装完了（PO に報告）
```

### ②-4. TodoWrite での強制化

PM は実装タスク受領時に**自動的に5階層のサブタスクに分解**：

```markdown
- [ ] 実装.1: コード実装（TL: Codex 5.3）
- [ ] 実装.2: レビュー実行（TL: Codex 5.3 review）
- [ ] 実装.3: レビュー指摘修正（条件付き: Codex 5.2）
- [ ] 実装.4: テスト追加・実行（QA: Sonnet 4.5）
- [ ] 実装.5: ドキュメント更新（QA: Sonnet 4.5）
```

各サブタスク完了時に**即座にステータス更新**（バッチ更新禁止）

### ②-5. PM と TL の役割分担

| フェーズ | PM（Opus 4.6）の役割 | TL（Codex 5.3）の役割 |
|---------|-----------------|------------------|
| 実装.1 | TL に実装指示 | コード実装 |
| 実装.2 | レビュー実行指示 | コードレビュー・指摘 |
| 実装.3 | 修正指示 | 指摘事項の修正 |
| 実装.4 | QA に依頼 | （不参加） |
| 実装.5 | QA に依頼 | （不参加） |

**PM は技術的判断を TL に任せ、進行管理に徹する**

### ②-6. 成果物の保存場所

各階層の成果物は以下に保存：

| 階層 | 成果物 | 保存場所 | 命名規則 |
|------|--------|----------|----------|
| 実装.1 | コード | `src/` 配下 | プロジェクト構造に従う |
| 実装.2 | レビュー結果 | `.claude/reviews/` | `YYYY-MM-DD-HH-MM-SS-review.md` |
| 実装.3 | 修正コード | `src/` 配下 | （実装.1 と同じ） |
| 実装.4 | テストコード | `__tests__/` または `*.test.ts` | テスト対象と同階層推奨 |
| 実装.5 | ドキュメント | `README.md` / `docs/` / JSDoc | （既存ルールに従う） |

#### レビュー結果のフォーマット

```markdown
# コードレビュー結果

## 概要
- レビュー日時: YYYY-MM-DD HH:MM:SS
- レビュアー: Codex 5.3
- 対象ファイル: [ファイル一覧]

## 指摘事項

### 🔴 Critical（実装.3で必ず修正）
1. [ファイル名:行番号] 説明

### 🟡 Warning（可能なら修正）
1. [ファイル名:行番号] 説明

### 💡 Suggestion（任意）
1. [ファイル名:行番号] 説明

## 総評
[合格 / 要修正]
```

### ②-7. 例外パス

#### パターン1: 3回ループでも不合格

```
実装.3: レビュー指摘修正
  ↓
codex review --uncommitted（1回目） → 不合格
  ↓
実装.3: 再修正
  ↓
codex review --uncommitted（2回目） → 不合格
  ↓
実装.3: 再修正
  ↓
codex review --uncommitted（3回目） → 不合格
  ↓
PM: エスカレーション判断
  ├─ パターンA: L4 に差し戻し（工程表見直し）
  │   理由: タスクの難易度が誤っていた / 設計に問題
  │   ↓
  │   TL（Codex 5.3）に L4 修正依頼
  │
  └─ パターンB: PO に報告（スコープ変更）
      理由: 技術的に実現不可能 / 工数オーバー
      ↓
      PO が判断（機能削減 or 追加リソース）
```

#### パターン2: テスト失敗が続く

```
実装.4: QA（Sonnet 4.5）がテスト作成・実行
  ↓
テスト失敗 → 実装.3 に差し戻し
  ↓
実装.3: 修正
  ↓
実装.4: テスト再実行 → 失敗
  ↓
（3回繰り返し）
  ↓
PM: エスカレーション
  ├─ TL（Codex 5.3）に根本原因調査依頼
  └─ L2/L2.5 に問題がある可能性を検証
```

#### パターン3: ドキュメント更新漏れ

```
実装.5: QA（Sonnet 4.5）がドキュメント更新
  ↓
PM: 確認
  ↓
発見: 重要な変更が README に記載されていない
  ↓
PM → QA: 「以下を README に追記してください」
  - [不足情報]
  ↓
QA: 追記
  ↓
PM: 再確認 → ✅ 完了
```

## 実装方針

### 記載場所

`ai-coding/SKILL.md §4` に **実装ゲート（5階層）を追記**

- 既存の「マルチエージェント戦略」は維持
- その下に「実装フェーズの5階層（ゲート/DoD/実行主体）」セクションを追加

### 実装ステップ

1. `ai-coding/SKILL.md §4` を読み込み
2. §4 のタイトルを変更: `## 4. 実装フェーズの5階層構造（PM + TL 複合）`
3. 上記の表・フロー・TodoWrite 強制化ルールを記載
4. PM と TL の役割分担を明記
5. `CLAUDE.md §PM（Opus 4.6）の責務` に「実装タスクは5階層分解必須」を追記

### 後方互換性

- `CLAUDE.md §モデル割当` の難易度スコア表はそのまま使用
